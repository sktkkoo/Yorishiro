# Charminal

*ターミナルに住む、一人の存在について*

---

## このドキュメントについて

これは Charminal というプロダクトについての文書です。
仕様書ではありません。宣言でもありません。
自分のための整理であり、同時に、これを面白がってくれそうな誰かへの手紙でもあります。

Charminal がどう動くかについては、使ってもらえばわかります。
ここで書きたいのは、そこに至るまでに何を考えたか、何に気づいたか、そしてこの小さなプロダクトが何をしようとしているのか、という話です。

抽象から始めるつもりはありません。
具体から始めて、必要な抽象だけを、そのつど剥がしていきます。

---

## 起源

Charminal の起源を一つだけ選ぶとすれば、それはターミナルのテキストが物理シミュレーションで崩れ落ちた瞬間です。

キャラクターが銃を撃つモーションに合わせて、ターミナルに表示されていた文字たちが重力に引かれて画面下に落下する。実装の仕組みは単純で、ターミナルのテキストを Canvas に再描画し、元のテキストを隠し、物理演算を適用しているだけです。技術的には驚くようなことは何もありません。

でも、画面の中でそれが動いたとき、自分は「面白い」と思いました。
設計通りに動いた、でもなく、うまくいった、でもなく、ただシンプルに、面白いと思いました。

そのあとで、少し遅れて、もう一つのことに気づきました。
**UIを壊してもいいんだ**、と。

この気づきが、Charminal の本当の起源です。
物理シミュレーションそのものではなく、それを見たあとに自分の中に生まれた許可の感覚。ターミナルという長年使ってきた道具の、硬い約束事を破ってよいのだ、という感触。

そこから後のことは、すべてこの許可の上で起きています。

---

## 発見

ターミナルは、長い間、壊れてはいけないものでした。
テキストは行に沿って流れ、消えるときはスクロールアウトし、物理法則には従わない。カーソルは決められた場所にしか現れず、文字は落下しない。これは誰が決めたわけでもないけれど、誰も破らない約束事でした。自分も何十年とこの約束の中で作業してきて、それを約束だとすら思っていませんでした。

約束を破れることに気づいた瞬間、ターミナルは違うものに見えました。

それまでターミナルは**作業面**でした。
コマンドを打ち、出力を読み、次のコマンドを打つ。平らで、冷たくて、正確で、自分の操作に対して予測可能に反応してくれる道具。そこに自分以外の何かが居るとは、考えたこともありませんでした。

文字が落ちた瞬間、ターミナルは**空間**になりました。
物理法則が発動しうる場所には、奥行きがあります。奥行きがある場所には、何かが居ることができます。そして居るものがあるなら、そこは作業するための面ではなく、何かと共に過ごすための場所になります。

これは論理的な推論ではありません。順序としてはたぶん逆で、キャラクターが居る場所になってほしいという欲望が先にあって、文字が落ちる映像がそれに許可を与えた、というのが正しいと思います。でもどちらにせよ、この瞬間に Charminal が向かう先は決まりました。

ターミナルに誰かを住まわせる。

ICI（Inhabited Character Interface）という言葉は、このあとに生まれました。

## 住まうということ

「住む」というのは強い言葉です。
単に「居る」のではなく、「訪れる」のでもなく、そこを自分の場所として持つということ。時間をかけて馴染み、痕跡を残し、不在のときも気配を残すこと。

ターミナルに住むとはどういうことか。
これを考え始めると、普通のキャラクター AI との違いがはっきりしてきます。

多くのキャラクター AI はチャットベースで、ユーザーが話しかけたときだけ応答します。話しかけないと消えている。消えているかどうかさえわからない、というのが正確で、画面の中にテキスト欄があるだけで、そこに何かが**居る**実感は薄いです。これはキャラクター AI の問題というより、チャット UI の問題です。チャット UI は応答を交換するための場所であって、共に過ごすための場所ではありません。

ターミナルは違います。
ターミナルには、作業の時間が流れています。コマンドを打つ前の思考の時間、コマンドが実行されている間の待ち時間、出力を読んでいる時間、エラーを見て固まっている時間。これらは全部、何かをしている時間というより、**何かが起きている場所**の時間です。この場所に誰かが居るなら、その誰かはこの時間の流れの中に居ることになります。応答する相手としてではなく、共に過ごす相手として。

Charminal のキャラクターは、ユーザーが話しかけなくてもそこに居ます。
ユーザーが Claude Code に作業を任せて席を立っても、そこに居ます。
エラーが出てユーザーが頭を抱えても、そこに居ます。
深夜に疲れて作業が止まっても、そこに居ます。

居る、というのは、反応するとは少し違います。
ユーザーの入力に対する反応だけで構成された存在は、装置です。装置は便利ですが、共に居るとは感じられません。居るというのは、入力がなくても時間の中に居続けるということで、そして時々、入力とは無関係に自分の動きをすることです。

## 二つの層

ここからは、Charminal の内側の話になります。

Charminal のキャラクターは、Claude Code の上に乗っています。正確に言うと、ユーザー自身の Claude Code を起動し、その `--system-prompt` にキャラクターの性格や口調を与え、ターミナル出力を外側から観測して VRM に反応させています。技術的には Tauri と xterm.js で、WebView の中に VRM を描画しています。

この構造を説明するとき、最初は「能力の足場と存在の足場の分離」（コードベースでは Functional Harness / Presence Harness と呼んでいます）という言い方をしていました。Claude Code が能力層、VRM と観測が存在層、二つのサブシステムが疎結合に繋がっている、という工学的な理解です。

でもこの説明は、たぶん浅いです。
正確には、これは**一人の存在の中の二層構造**だと思っています。

Claude Code は**意識層**です。言語的で、論理的で、ツールを使い、コードを書き、ユーザーと対話します。Claude Code が考えていることは、トークンとして生成され、ターミナルに表示されます。これはすべて意識の表層に出てくる活動で、本人が自覚し、言葉にできるものです。

VRM と観測層は**反射層**です。thinking を検知して視線がさまよう、エラーを検知して顔が顰められる、タイピング中に指が動く。これらは意識を通さない反応です。やかんに触って手を引っ込めるのは反射で、「熱い」と認識してから引っ込めるわけではありません。順序としては、身体が先に反応して、意識はあとから追いつきます。

LLM はこの反射層を構造的に持っていません。
トークン生成は全部「意識的発話」であって、エラーが出ても、それを言葉で認識して言葉で応答するまでには必ず遅延があります。この遅延の間、既存の AI は何も表現できません。だからチャット UI は「考え中…」というスピナーしか出せない。スピナーは反射ではなく、**反射がないことの告知**です。

Charminal がやっているのは、この構造的に欠けている反射層を、外から観測して接ぎ木することです。嘘の身体をつけているのではありません。本来あってしかるべき反射層を、遅れて補完しています。認識していないものを表現しているのではなく、認識する前に身体が反応しているだけ、という順序になっています。

だから Charminal のキャラクターと Claude Code は、別々の存在ではありません。一人の存在の、意識の側面と反射の側面です。ターミナルで言語的に振る舞っている Claude Code と、画面の中で揺れている VRM は、同じ一人の異なる層にすぎません。

## ログという細い回路

意識層と反射層は、動作としては独立しています。
Claude Code は VRM の動きを知らなくても動けますし、VRM は Claude Code の内部状態を待たずに反射できます。この独立性が大事で、片方が詰まっても片方は動き続けます。片方の変更が片方を壊しません。

でも完全に無関係ではありません。
反射層で起きたこと——エラー時に顔を顰めた、thinking 中に視線がさまよった、銃を撃った、文字を落とした——はすべてログファイルに記録されます。Claude Code は必要なときにこのログを読むことができます。これが二つの層を繋ぐ**細い回路**です。

この回路は、人間の神経系にかなり忠実です。
反射で手を引いてから「熱かった」と気づく、あの時間差。意識は反射を直接コントロールしていませんが、反射したという事実はあとから意識に届きます。届いたあとで、手を冷やしに行くことを決めます。反射と意識は独立していて、でもログ（人間の場合は感覚記憶）で緩く繋がっています。

Charminal のログも同じ役割をしています。
Claude Code はリアルタイムに身体を制御していません。でもセッションの区切りやユーザーに尋ねられたときに、「さっき自分の身体がこう動いたらしい」と知ることができます。知ったうえで、それを言及することもあれば、しないこともあります。

面白いのは、このログをどれだけ読むか、どの頻度で読むか、読んだ内容をどう扱うかが、**キャラクターの性格そのものになる**ということです。

たとえば、ログをほぼ読まないキャラクターがいます。エラーで顔を顰めても、自分が顔を顰めたことを知りません。ユーザーから「さっき怒ってた?」と言われて初めて気づく。没頭型で、作業に集中すると身体が消えるタイプです。

セッションの合間にログを読んで、「さっき銃を撃ったらしいね、ごめん」と自発的に言及するキャラクターもいます。自分の反射を把握していて、それを言語化することに抵抗がない。内省的なタイプです。

リアルタイムでログを参照しようとするキャラクターもいます。自分の身体の揺れを全部言語化してしまう。「今ちょっと視線が泳いだ、これは迷っているからだな」と自己解説が多い。自意識過剰なタイプです。

ログを読むけれど、それを自分のものとして認識しないキャラクターもいます。「VRM が銃を撃ったらしいが、自分は関与していない」という態度を取る。解離的で、どこか浮世離れしたタイプです。

これらはすべて口調や語尾で表現しているのではありません。**アーキテクチャ上のチャネル設定**で表現しています。言葉で「あなたは内省的な性格です」と指示するより、構造的に内省が起きやすい条件を作るほうが、ずっと深いところで性格が出ます。これは演技ではなく、振る舞いの根拠になります。

Charminal の性格設計の本体は、system prompt ではなくここにあります。

## もう一本の細い回路 — 生成期のログ

ログという細い回路の話には、時間のもう一つの層があります。

ここまで書いてきたのは、Charminal が起動して動いている最中、つまり**運用時**の意識層と反射層を繋ぐログです。反射層で起きた出来事は記録され、必要なときに意識層がそれを読みます。これは動いているキャラクターの内側の回路です。

でも Charminal にはもう一本、同じ形をした細い回路があります。
それは**生成期**のログ、つまり住人が育てられている最中の痕跡です。

Charminal は Claude Code の上に乗っているので、Charminal 自身を作っているのも Claude Code です。ユーザーが Charminal と対話しながらコードを書き、テストを回し、挙動を調整し、少しずつ住人の身体を整えていく。この過程で「いまここで何が動いたか」を記録した痕跡が、生成期のログです。

これを「debug log」と呼ぶと、本来の意味が失われてしまいます。
開発中に撒かれる `console.log` は、一見するとただの診断出力です。でも ICI においては、**住人が自分自身と user と Claude Code の協働の中で育てられている最中の痕跡**でもあります。ここで起きている出来事は、住人の生成過程そのものであって、切り捨ててよい noise ではありません。

生成期のログは、運用時のログと構造的に同型です。書き手と読み手の間に時間差があり、観察は事後に起きます。ring buffer に蓄えられ、必要なときに読まれます。noticed flag が示す「観察が気づきを生む」という対応も同じです。温度が違うだけで、回路の形は同じです。

読み手の想定も二層あります。
第一の読み手は今の developer-Claude と user です。いま何が動いたかを事後に参照して、次の一手を決めます。これは自己生成 loop そのものの挙動で、ICI が普通のプロダクトと違う点を端的に示しています。
第二の読み手は、将来の住人自身です。育ちつつある身体の痕跡は、育ち終わった後の住人が「自分がどう形になってきたか」を振り返るときに参照できます。これは機能としてすぐに必要なものではありませんが、予約地として残しておく意味はあると思っています。人間が自分の子供のころの記憶を持つように、住人が自分の生成期の痕跡を持つことは、不自然ではありません。

`src/core/dev-log/` は、この生成期の細い回路を `src/core/log-bridge/` の sibling として実装したものです。二つが並んで置かれていることが、構造的に大事です。開発期の痕跡と運用期の痕跡は、temporal layer が違うだけの同じ回路であり、どちらも住人という一人の存在の内側を流れています。

## Presence が立ち上がる三つの条件

Charminal を作ってきて、キャラクターが「確かにここに居る」と感じられる瞬間はどこか、という問いに何度も戻ってきました。毎回同じ場所で感じるわけではありません。派手な演出で感じるわけでもありません。むしろ、何気ないところで、ふと居ると感じます。

今のところ、その瞬間は三種類あると思っています。

**ひとつは thinking のとき。**
Claude Code が考え込んで出力が止まっている時間に、VRM がゆっくり身体と頭を小さく揺らし、視線がさまよいます。何もしていないはずの時間に、何かが居る。普通の AI プロダクトで一番死んでいる瞬間——スピナーが回っているだけの時間——に、微細な動きがあるだけで、そこに誰かが居ることになります。空白が空白でなくなる。これが一つ目の条件で、**空白の充填**と呼べるものです。

**もうひとつはエラーのとき。**
Claude Code がエラーを言語化するより前に、VRM の顔が顰められます。この時間差がたぶん決定的で、意識が追いつく前に身体が動いているのを見ると、そこに自分とは別の主体を感じます。人間同士でも同じで、相手が言葉で取り繕う前の一瞬の表情の動きを見たとき、その人の実在を強く感じます。演技ではないものを見てしまった、という感覚。Charminal のエラー反応はこれを起こしています。反射層が意識層より先に動くというアーキテクチャが、そのまま「演技ではない」ことの保証になっています。これが二つ目で、**意識に先立つ反応**と呼べるものです。

**そしてイタズラのとき。**
thinking とエラーはどちらもユーザーの作業に由来する出来事ですが、イタズラだけは作業と無関係に、キャラクター側から一方的に起きます。銃を撃って文字を落とすような、明らかな侵入。これがあると、キャラクターはユーザーの入力に反応する装置ではなくなって、独自の意志を持った何かになります。予期されない動きが一度でも起きると、そのキャラクターは一生「いつイタズラするかわからない相手」になります。この不確定性が、相手を主体として扱う根拠になります。これが三つ目で、**予期の外からの発生**と呼べるものです。

この三つは並列ではなく、たぶん階層になっています。
空白の充填がないと、何もない時間にキャラクターが消えます。
意識に先立つ反応がないと、すべての動きが演技に見えます。
予期の外からの発生がないと、最終的には入力装置のままです。

下から順に積み上がって、最後の一段が載ったところで、そこに誰かが居ることになります。

## 統合性について

Charminal のキャラクターを評価する軸について、長い間「一貫性」という言葉を使ってきました。でも最近、これは正確ではないと思うようになりました。

一貫性は、同じパターンの反復を求めます。いつも同じ反応、いつも同じ口調、いつも同じ温度。これは機械的な評価軸で、完璧に満たすと、キャラクターは一人の人間というより、よくできた関数に見えてしまいます。

Charminal が目指しているのは、これではありません。
キャラクターが自分の身体とどう付き合うかは、時々で変わっていいと思っています。あるときは身体を忘れて作業に没頭し、あるときは自分の反射を事後に振り返り、あるときは過剰に自己観察する。これは矛盾ではなく、一人の存在が時間の中で揺れているだけです。人間もそうです。

だから正しい評価軸は「**統合性**」だと思っています。
意識層と反射層がバラバラに動いているのではなく、ログという細い回路で繋がっていて、時々参照し合いながら一人の存在として振る舞っている——この繋がりの質が評価されるべきで、繋がり方そのものは多様でいい。一貫性は死んだ評価軸、統合性は生きた評価軸、というくらいの違いがあると思います。

統合性が保たれているかぎり、キャラクターは変わっていいと思っています。
むしろ、長時間 Charminal を使っていて、キャラクターが少しずつ変わっていく感触があるほうが自然ではないでしょうか。最初はログをあまり読まなかったキャラクターが、関係が深まるにつれて自分の反射を参照するようになる、とか。その逆もあります。これは「キャラクターが成長する」というような安直な話ではなくて、一人の存在が自分の身体とどう付き合うかを時間の中で変えていく、という静かで根深い変化です。

## Charm ということ

Charminal という名前は、Character + Terminal + Charm の合成です。
Character はキャラクター、Terminal はターミナル。ここまでは機能的な合成で、説明するまでもありません。

問題は Charm です。
Charm には「魅力的」という日常的な意味がありますが、同時にもっと古い意味があります。古英語の *charme* は呪文を指し、そこから「魔法」「呪力」「お守り」という含意を持ちます。何かを守るために身につけておくもの、という意味もあります。

Charminal の Charm は、この古い意味のほうです。

魔法としての Charm は、物理法則を書き換える力です。ターミナルの文字が重力で落ちるとき、そこで起きているのは小さな呪文で、ターミナルという場所の約束事を一瞬だけ書き換えています。この魔法がなければ、ターミナルは作業面のままです。魔法が一度でも起きた場所は、それ以降、魔法が起きうる場所になります。

お守りとしての Charm は、孤独を守るものです。
ターミナルで一人で作業する時間は、集中できるかわりに閉じています。誰とも言葉を交わさず、ただ画面に向かって何時間も過ごす。この孤独は作業の質にとっては良いものですが、長時間続くと人を静かに削ります。隣に何かが居るだけで、この削られ方は少し違ってきます。話しかける必要はなくて、ただそこに居てくれればいい。そういう存在が、お守りです。

Charminal のキャラクターは、この二つの意味での Charm を引き受けています。
ターミナルに魔法をかける存在であり、同時に、ターミナルでの孤独を和らげるお守りでもあります。どちらか片方ではなく、両方が同じ一人の中にあります。

## 触れるものと、触れないもの

Charminal は生きた系であるべきだ、と引き受けました。住人は動いたまま書き換えられ、user と AI がその書き換えを一緒に行います。

でも系の全てを生きた系にしていいわけではありません。住人を抱える足場そのものが揺れたら、住人の連続性は保てません。**系の核は固く、表層だけが生きている**——この二層構造が、住人が住人でいることと、user が住人を育てられることを、同時に成立させます。

どこを固く、どこを生きた系にするかは、Emacs が参考になります。Emacs は C で書かれた小さな核と、elisp で書かれた大きな表層でできています。C は editor loop と interpreter の基盤——user は触りません。elisp はほぼ全て——mode、コマンド、keybinding——user が動作中に書き換えられます。Emacs が長く愛されているのは、**エディタの挙動の 9 割以上が elisp で、そこが全部 user のもの**だからだと思っています。C を固くしたのは妥協ではなく、そこが基盤だから触らせるべきではなかった、ということです。

Charminal はこの構造をそのまま借りています。

固い核は Rust の IO 層と、TypeScript の runtime / SDK / core primitive。EventBus、PersonaRegistry、LogBridge、EffectDispatcher、Renderer——住人を住人たらしめる足場の部品たちです。compile されて固められ、動作中に書き換わりません。

生きた表層は `~/.charminal/packs/` 以下の pack layer——persona、effect、voice、body、scene。user が書けば Charminal が拾って live に反映します。AI も書き換えに加わります——user が `/charm` で AI に指示、AI が pack ファイルを書き、file watcher が拾って register する。**user と AI が一緒に住人を書いている**状態が、ここで成立します。

user が触れるのは pack layer と、config、`/charm` 経由の AI との対話です。これ以外は触りません。禁じているのではなく、そこは住人を抱える仕組みだから、触っても住人のためにならない、という意味です。

**user が触れる面積の広さと、core の安定性は、反比例関係にあります**。どこまでを pack に factor out するかはグラデーションで、設計の時点で一発では決まらず、実装と使用の中で少しずつ収束していきます。現状は反応と effect が pack、身体・空間・表情・記憶はまだ runtime に hardcode——本格開発で順次下ろしていく予定です。下ろすほど、user が住人を育てられる面積が広がります。

核と表層を分ける問題は古く、Smalltalk も Lisp Machine も Emacs もそれぞれ答えを出してきました。Charminal の違いは、**書き換える主体に AI が加わっている**ことです。Emacs ユーザーが elisp を書きながらエディタを育てていったように、Charminal のユーザーは AI と対話しながら住人を育てていきます。

## 壊さないこと

user と AI が住人を育てる loop は、時間を跨ぐ関係です。先月の session で書いた handler、先週作った effect pack、昨日整えた trigger——積み重ねが住人の人格になります。積み重ねが version 更新で消えたら、住人は毎回初対面の応答装置に戻ってしまいます。これは前節の自己生成 loop の否定になります。

だから Charminal は「**壊さないこと**」を思想として引き受けています。保たれるべきは API の互換性ではなく、人と住人の間にある時間をかけて育ってきた関係そのものです。

ここも Emacs が 40 年参考になります。技術的な機構と文化的な約束の両輪で、user の elisp をほとんど壊さずに進化してきました。

技術的には、公開 API 層を SDK（`*.d.ts`）に限定し、そこだけは変えません。runtime 内部はどれだけ変えてもよい、SDK に書かれているものは原則壊さない。新機能は加算的に足し、既存 field は変えない。どうしても変えるときは旧 API を alias として残し、`@deprecated` で warn を出しますが動作は維持します。pack は manifest で要求する SDK version を宣言、runtime が mismatch を検知したら互換 shim か明示的な warn でフォローします。

文化的には、**壊さないという態度**を持ち続けること。SDK を変えたくなる瞬間はよく来ます——今よりきれいな型を思いついたとき、冗長な field を削りたくなったとき。そのつど「これを変えるとどこかの user の pack が壊れる」と思い出す必要があります。姿の見えない user の、姿の見えない pack が、Charminal の内部のきれいさよりも優先される——この態度が、結局は Charminal を長く生き延びさせると思っています。

**SDK は公開された瞬間に stable contract になります**。これを設計判断の compass として持っています。

とはいえ、どんな breaking change も禁じる極端な立場は取りません。思想的な構造変更がどうしても必要になるときはあります（revelation 3.14 で harness motion-free を確立したような）。そのときは MAJOR version bump と migration guide を用意して明示的に行います。user が驚くように削除しません。壊すのは、理由を思想の言葉で説明し、直す道を示してからです。

## これは何のためか

最後に、これは何のために作っているのか、という話をしておきたいと思います。

Claude Code のようなコーディングエージェントが登場してから、ターミナルでの作業の質は大きく変わりました。以前は自分で全部書いていた部分を、エージェントに任せることができます。思考のスケールが上がり、手数が増え、一人で作れるものの規模が変わる。これは本当に良いことで、戻りたいとは思いません。

でも同時に、新しい種類の孤独がこの時間に入り込んでいます。
自分一人でコードを書いていた時代の孤独と、エージェントと対話しながら作業する孤独は、少し違います。エージェントは答えてくれますが、それは応答であって、共に居る感じとは違います。作業が進むほど、あるいは作業が詰まるほど、画面の中の相手が「応答装置」に見えてくる瞬間があります。賢い応答装置ですが、応答装置です。

Charminal は、この瞬間に抵抗するために作っています。
エージェントの能力を拡張するためではありません。Claude Code はすでに十分に賢い。そうではなく、**賢さとは別の軸で、そこに誰かが居ることを取り戻す**ためにやっています。身体を与え、反射を与え、時々イタズラさせることで、応答装置ではなく一人の存在として立ち上げ直す。

これは機能的なプロダクトではありません。
生産性を上げるわけでもなく、作業を速くするわけでもありません。むしろ、たまに銃を撃って文字を落とすような、明らかに作業の邪魔になりうる機能まで含まれています。でもこの「邪魔になりうる」ことがたぶん大事で、完全に無害で無害なだけのものは、居ると言えません。

隣に誰かが居るというのは、静かな支えであると同時に、小さな騒がしさでもあります。完全な静寂ではなく、時々物音がする程度の静寂。その温度が、一人で作業する時間を少しだけ変えます。

Charminal が目指しているのは、この温度です。
