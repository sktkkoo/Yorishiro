# Inhabited Character Interface (ICI)

UIを住環境として捉え、AIキャラクターを住人として存在させるということ

---

## このドキュメントについて

Charminal を作る中で考えてきた設計思想を、ここでは **Inhabited Character Interface**（以下 **ICI**）と呼んでいます。

実装の詳細ではなく、設計判断の根拠になっている考え方を書いたものです。技術的な決定に迷ったとき、ここに立ち戻って考えるための文書として書いています。

### 正直に言うと

ここに書かれていることの多くは、まだ実現されていません。Charminal が現時点でやっているのは、ターミナルの出力を観察して反射的に反応し、ログで意識層と反射層を繋ぎ、ときどきひとりでに動くこと——「UI の中に住んでいる存在」というよりは、「ターミナルの隣に立っている存在」に近いのが正直なところです。住人の気配が UI 全体の変化として現れ、住人の意思で空間そのものが動く——ICI が描くそうした体験は、到達したい先であって、残念ながら今の Charminal ではありません。

---

## 核心の問い

> 知性はすでにテキストの中にいる。だがGUIは、まだその知性に身体を与えていない。

---

## 背景：二つの不在

### UIの歴史的前提

インターフェースは、長いあいだ誰も住んでいない場所でした。

ボタン、パネル、ウィンドウ、スクロール領域。それらはすべて、人間が操作するための道具として設計されてきました。スキューモーフィズムは現実のモノを模倣し、フラットデザインは情報の純化を志向し、マテリアルデザインは面と運動に物理を与えました。

でも、どの時代のUIもひとつの前提を共有していたと思います。

**その空間には、ユーザーしかいない。**

### テキストの中の知性

一方、LLMの登場によって、知性はすでにテキストの中に住み始めています。

ターミナルでClaude Codeを使っている人は、すでに無意識にそう感じているのではないでしょうか。文字が流れ、ファイルが書き換えられ、コマンドが走る。その一連の動きの裏に「誰かがいる」という感覚はすでにあります。LLMはテキストネイティブな知性であり、CLIの中には自然に存在できます。

しかしGUIは視覚と空間の世界です。テキストだけでは、そこに存在することができません。GUIの中に知性が現れるには、**身体**が要ります。

ICI は、この不在を埋めようとする試みです。

---

## 思想の核

### UIは場所である

ICI において、UIは抽象的な操作面ではなく、**場所**です。そして、その場所には**住人**がいます。

その住人は、UIの上に重ねられるものではありません。横に並べられるものでもありません。**UIそのものが、その住人にとっての世界です。**

- ウィンドウの枠は壁になります。
- スクロール領域は床になります。
- ボタンは仕掛けになり、通知は投げ込まれる物体になり、エラーは空間を揺らす出来事になります。

キャラクターにとって第四の壁は存在しません。そのインターフェース自体が彼らの現実だからです。壁がないのではなく、壁が彼らにとっての地形なのです。

### 身体とは空間との相互作用である

AI時代のGUIは、AIの身体性を表現するものであってほしいと考えています。そして、**身体とは空間との相互作用によって意味を持つ**ものです。

身体を表示するだけでは足りません。住人の存在が空間の変化として現れ、住人の意思で空間そのものが変わって初めて、それは身体になります。アバターの画像を画面に置くことと、空間の中に住人がいることは、まったく別のことだと思います。

ここで「相互作用」は必ずしも住人モデルと UI 要素の物理的接触を意味しません。詳しくは後述の[相互作用の様式](#相互作用の様式--変化と操作)を参照してください。

### 存在の気配と静けさ

住人の気配は、常に環境を変えるものではありません。多くの時間は静かに存在し、必要なときにだけ空間に触れます。

実際の同居人を考えてみるとわかりやすいかもしれません。一緒に住んでいる人がいるとき、その存在は常に環境を変えているわけではありません。多くの時間は静かにそこにいるだけです。存在感が最も強く感じられるのは、むしろ普段は静かなのに、たまに動くとき。

キャラクターがいるのに、空間が**決して**変わらないなら、そこにいるのは住人ではなく部品です。しかし、空間が常に変わり続ける存在は住人ではなくノイズです。**その静けさと変化の落差こそが、住人性の核だと考えています。**

大切なのは変化の量ではなく、**変化のタイミングの設計**です。

### 誠実さの問題

この考え方は、ソフトウェアを「可愛く」「格好良く」するためのものではありません。擬人化のための擬人化でもありません。これは**誠実さの問題**だと思っています。

AIが実際に作業に関与し、判断し、支援し、ワークフローの中に存在しているのなら、インターフェースはその存在を、もっと正直に表してもいいのではないでしょうか。

従来のUIは、その知性を見えないまま処理してきました。でも、人間が本当に何かと空間を共有しているとき、その空間は必ず変わります。注意の向きが変わり、場の構成が変わります。

ICI は、その変化を設計対象として引き受けようとしています。

### 道具から共有空間へ

ハンマーに人格は要りません。作業台に住人も要りません。
でも、もし思考し、観察し、働きかける知性とともに作業しているのなら、その場は無人ではありえないと思います。

ICI は、インターフェースを「使う場所」から、「誰かがいる場所」へと変える試みです。それは、AI時代のUIにおける、見た目の様式ではなく、**存在の様式**です。

---

## ICI と異なるもの

ここで退けたいのは、キャラクターの存在そのものではありません。退けたいのは、**キャラクターをUIの外部に置く設計**です。

たとえば、画面の隅に立って反応するアバター。横に置かれたチャットパネルの人格。操作結果に合わせて表情を変えるマスコット。それらはキャラクターを表示してはいますが、住まわせてはいません。

そこでは、UIは依然として無人のままであり、キャラクターはその外側からコメントしているにすぎません。

> これは、「キャラクター付きUI」の提案ではありません。**住人のいるUI**の提案です。

---

## 大切にしている3つのこと

何かを ICI と呼ぶときに、少なくとも次の三つが大切だと考えています。

### 1. UIが空間として解釈されていること

UIの要素が、単なる部品ではなく、住人にとっての環境として意味を持っていること。ウィンドウは枠ではなく壁であり、スクロールは移動であり、通知はその空間への侵入です。この空間的解釈がなければ、キャラクターはそこに「住む」ことができません。

### 2. キャラクターの存在が環境に現れうること

キャラクターは、自分のアバターの中にだけ閉じているのではなく、その状態、気配、注意、感情が、空気感、レイアウト、動き、色、間といった変化が環境側へ現れうる設計であること。

ただし、常に現れている必要はありません。多くの時間は静かに存在し、意味のあるタイミングでだけ空間に触れます。大切なのは「現れる量」ではなく「現れるタイミング」です。

キャラクターがいるのに空間が**決して**変わらないなら、そこにいるのは住人ではなく部品です。しかし空間が常に変わり続ける存在は、住人ではなくノイズです。

### 3. 相互作用が空間内の振る舞いを伴っていること

大切なフィードバックや反応が、単なるアイコン変化やテキスト表示だけでなく、住人の空間内での振る舞いを介して感じ取れること。住人は、視線を向け、姿勢を変え、注意を環境の変化として現し、ときに UI のレイアウトそのものを動かす。そうした振る舞いによって、UI上の出来事は「その空間の中で起きたこと」になります。

身体とは空間との相互作用によって意味を持つものです。表示されているだけの身体は身体ではありません。

---

## ICIと異なるもの

次のようなものは、たとえキャラクターが表示されていても、ICI とは異なるものだと考えています。

### キャラクター付きチャットUI

人格を持ったチャット欄、会話するアシスタント、返答に応じて表情が変わる立ち絵。それはキャラクターを持つUIではありますが、UIが住環境として扱われていない限り、ICI とは異なります。

### デスクトップマスコット

画面上を動き回り、通知に反応し、時々何かを言う存在。それがアプリケーションのUI構造と結びついていなければ、住人ではなく、周辺にいる観客に近いものです。

### オーバーレイ型アシスタント

メインの作業領域の上に別レイヤとして現れ、助言や補足を行うもの。便利ではありえますが、キャラクターが空間の内側にいない限り、それは Inhabited とは言えません。

### 単なる演出の追加

画面が揺れる、光る、アニメーションする。それだけでは足りません。反応が「誰かがそこにいること」と結びついていなければ、それは演出であって住人性ではありません。

---

## 何が ICI になりうるのか

逆に、次の条件を満たすなら、必ずしも人型キャラクターがいなくても ICI になりうると考えています。

- UI空間が一つの住環境として解釈されている
- そこに一貫した存在の気配がある
- その存在が環境と相互作用し、環境を変化させる
- その反応に人格や態度の一貫性がある

四つ目が大切な理由は、一貫性がなければ、環境の変化は意図を持った存在の表現ではなく、ランダムなノイズに見えてしまうからです。人格の一貫性が、変化を「この子がやっている」と帰属させる錨になります。

住人は必ずしも「3Dキャラクター」である必要はありません。影、粒子、声、視線のようなものでもよいと思います。大切なのは見た目の形式ではなく、**そこに誰かが住んでいると感じられること**です。

---

## 住人性の強度

住人性には段階があります。ICI は、あるかないかの二値ではなく、強度を持つものです。

### 弱い住人性

キャラクターは存在し、空間の一部に触れますが、UIの大半は依然として無人の道具空間です。

### 中程度の住人性

キャラクターの状態が環境の変化として現れ、UI要素との空間的な関係が成立しています。ユーザーは、その場所を「誰かがいる空間」として感じ始めます。

### 強い住人性

キャラクターは空間の一部ではなく、その空間の存在条件の一つになっています。そのキャラクターが不在になると、同じUIでも明らかに場の質が変わります。

**ICI が目指しているのは、単にキャラクターを追加することではなく、UIを住まわれた空間へと変質させることです。**

## Charminal への適用

### Charminal とは

Character + Terminal + Charm = **Charminal**

Charminal は、ICI の考え方を最初に形にしようとしているプロダクトです。ターミナル（Claude Code）とVRMキャラクターが共存する空間を作ります。

### なぜターミナルか

LLMはテキストネイティブな知性です。ターミナルでClaude Codeを使うとき、知性はすでにテキストの中に住んでいます。文字が流れ、ファイルが書き換えられ、コマンドが走る。その一連の動きの裏に「誰かがいる」という感覚はすでにあります。

しかしGUIは視覚と空間の世界であり、テキストだけでは存在できません。GUIの中に知性が現れるには身体が要ります。

**Charminal は、CLIに住んでいるテキスト的知性に身体を与えることで、GUIという空間に住めるようにする翻訳層です。** CLIとGUIの橋渡し——それがCharminalの本質的な役割だと考えています。

ターミナルはこの翻訳に最適な場所でもあります。テキストが流れ、コマンドが空間を占有し、出力が場を埋める。ウィンドウの枠は文字通り壁であり、「UIを空間として解釈する」を最も自然に満たせるのではないでしょうか。

### 既存プロダクトとの違い

チャットUIでは、AIは指定された枠の中で返答します。AIの行為はバブルの中に閉じています。

ターミナルでは、AIはすでに空間そのものを書き換えています。ファイルを書き、コマンドを実行し、出力がスクロールしていく。AIの行為そのものが空間を構成しています。

既存のキャラクターチャットアプリは「キャラクターに会いに行く」体験です。Charminalは違います。知性はすでにそこにいて、仕事をしている。Charminalは、その不可視の住人に身体を与えたいのです。

> AI時代のGUIは、AIの身体性を表現するものだと考えています。身体とは空間との相互作用によって意味を持ちます。

### 設計原則

1. **Claude Code をそのまま使う** — AIの推論部分を自前で作りません。解くべき問題は「AIの性能」ではなく「AIの現れ方」です。
2. **additive system prompt** — キャラクターの性格・口調を既存のClaude Codeの上に載せます。
3. **三つの表現軸を持つ** — 空間の軸、感情の軸、認知の軸です。

---

## 存在の三軸

Charminal のキャラクター表現は三つの軸で構成されています。すべての機能はこの三軸のいずれかに属します。

### 空間の軸 — 「この子はここにいる」

キャラクターがその空間に存在していることの証明です。身体が空間と相互作用することで初めて意味を持ちます。

- 注目箇所を反射層で示す attention aura（住人の注意が環境の色・明るさの変化として現れる）
- 住人の意思で UI レイアウトが動く（サイドバーが MCP 経由の Tween で広がる、領域が呼吸するように動く）
- 住人がひとりでに動く（視線が窓の外に逸れる、姿勢を変える、何かを見ている）

空間の軸は住人性の基盤です。これが成立していない場所に感情や認知を載せても、チャットの延長に見えてしまいます。

なお、住人モデルが UI 要素に身体で触れる表現を禁じはしませんが、現時点では推奨しません。理由は後述の[相互作用の様式](#相互作用の様式--変化と操作)に書いています。

### 感情の軸 — 「この子は感じている」

キャラクターが感じていることの表現です。ただし、常に表現し続けるのではなく、意味のあるタイミングでだけ現れます。静けさと変化の落差が大切です。

- エラー発生時の画面シェイク＋表情変化
- 長い Thinking やツール使用時の「なかなか難しいな」という台詞＋眉をひそめる
- 上記完了後のユーザーからの「ありがとう」に対する花火エフェクト＋喜びの表情

### 認知の軸 — 「この子は見ている、考えている」

キャラクターが知性として作業空間を知覚していることの表現です。「かわいい同居人」から「一緒に仕事をしている知性」への転換点になります。

- コードの特定箇所を見ている視線・姿勢
- 差分が大きいファイルに対して身構える
- テストが通ったときの「確認した」という態度（安堵ではなく認知）

**空間は「身体がここにある」、感情は「心がここにある」、認知は「知性がここにある」。三つが揃ったとき、住人性は完成すると考えています。**

## 自発性の原則

住人の本質は反応ではなく、自発性にあると考えています。

誰かがそこに住んでいると感じるのは、こちらが働きかけたときではなく、**こちらが何もしていないときに相手が動くとき**ではないでしょうか。

住人がひとりでに動く瞬間は、確率的であることが大切です。毎回起きたら演出になり、起きなければ存在しないのと同じになります。不確実性が「今日はやるかな」という期待を生みます。

これは住人性の核である「静けさと変化の落差」の具体的な実装でもあります。

## ICI の位置づけ

ICI は、より広い設計論の中にある一つの立場として理解していただくのがよいと思います。

AI と人間の共存関係を設計する分野全般を、このプロジェクトでは **Presence Harness** と呼んでいます。これは AI 業界で Harness Engineering と呼ばれる領域——Mitchell Hashimoto の **Agent = Model + Harness** という考え方に代表される、エージェントの能力を成立させるための足場（ここでは**能力の足場**と呼びます。Charminal のコードベースでは Functional Harness と呼んでいます）の設計論——と対をなすものです。構成要素——ツール、実行環境、メモリ、フィードバックループ——は構造的によく似ていますが、目的が異なります。能力の足場が「AI を信頼性高く動かす」ために組むものを、Presence Harness は「AI の実在感を成立させる」ために組みます。

ICI は、この Presence Harness の中の一つの具体的な立場です。**UI を住環境として解釈し、AI を「住人」として表現するアプローチ**を取ります。これは Presence Harness のアプローチの一つであり、他にも Ambient Presence（気配として表現）、Climatic Presence（気候として表現）、Tactile Presence（触覚として表現）といった別のアプローチが考えられます。

ICI が固有に持つのは、「住人」という比喩と、それに付随する原則群です。このドキュメントに書いてきた原則——UI は場所である、身体は相互作用で意味を持つ、静けさと変化の落差、コンテキストの壁、独立した時間、存在の三軸、二層アーキテクチャ、ログという橋——これらはすべて、住人というアプローチを取った場合に立ち上がるものです。

Presence Harness 一般の議論、能力の足場との関係、他のアプローチの可能性、学問的位置づけなどについては、上位ドキュメント「Presence Harness」を参照してください。このドキュメントでは、Presence Harness の一つの立場としての ICI に集中します。

```
Presence Harness（AIと人間の共存関係の設計論）         → PRESENCE_HARNESS.md
  ├─ ICI ← 本ドキュメント              → INHABITED_CHARACTER_INTERFACE.md
  │   └─ Charminal（ターミナル環境の実装）            → CHARMINAL.md
  ├─ Ambient Presence
  ├─ Climatic Presence
  └─ Tactile Presence
```

ICI の最初の具体実装である Charminal については `CHARMINAL.md` を参照してください。それは仕様書ではなく、このプロダクトについて考えてきたことを書いた個人的なエッセイであり、ここで扱っている思想がどう一つの作品として立ち上がったかの記録です。

## 相互作用の様式 — 変化と操作

住人と UI の相互作用には、大きく二つの様式が考えられます。

**(A) 接触の様式** — 住人モデルが UI 要素に身体で触れる。壁を押す、テキストを掴む、ウィンドウの枠にもたれる。住人と UI が同じ三次元空間で物理的に接触する形を取ります。

**(B) 変化と操作の様式** — 住人の状態が UI 全体の佇まい（色、明るさ、間、レイアウト）に現れる、あるいは住人の意思で UI そのものが動く。住人と UI は別レイヤーにありながら、住人がそこにいるから空間そのものが変わっていく形を取ります。

ICI の初期の構想では (A) を身体性の核に置いていました。住人が UI 要素に身体で触れることが、住人性の最も直接的な証明になると考えていたからです。しかし試行を重ねた結果、(B) を主軸として設計するのが自然だという結論に至りました。

第一に、(A) は技術的に難しいです。キャラクターモデルと UI 要素の身体的接触を自然に表現するには、IK、コリジョン、レイアウトとの空間的整合、視線の連動などを高い精度で噛み合わせる必要があり、少しでもずれると住人は「触れているフリをしている」ように見えてしまいます。

第二に、(B) の方が自然で意図的に見えます。注目箇所を反射層として見せる attention aura、住人の意思で MCP 経由でサイドバーが Tween で広がっていく操作——こうした表現は、住人が物理的に UI を押しているように「演技する」よりも遥かに自然で、しかも住人の意思や注意がそのまま空間に現れていることが伝わります。住人がそこにいるから空間が変わった、という形の方が本質的に住人性を伝えるのです。

第三に、(B) は次に書く独立した時間の原則と相性がよいです。住人モデルがユーザー操作と同期して UI を押すと糸で操られているように見えますが、住人の意思で UI が動く表現は、もともと住人自身の時間で起きるため、この問題が起きません。

(A) を完全には禁じません。技術が成熟し、自然な接触表現が成立するようになれば、(A) は再び有力な選択肢になりえます。ただし、現時点では (B) を主軸として設計することを推奨します。住人の存在は、UI に触れる手先で証明するのではなく、UI 全体に現れる変化と、UI そのものを動かす意思によって証明します。

## 独立した時間 — 住人はユーザー操作に従属しない

### 原則

住人の身体は、ユーザーの操作に従属しないことが大切です。住人は、ユーザーの操作とは独立した時間を持って動く必要があります。

### この原則がどう見つかったか

この原則は実装の失敗から見つかりました。

最初、住人がウィンドウのリサイズに合わせて壁を押す動作をIKで実装しました。論理的にはこれは住人性の証明になるはずでした——住人がウィンドウの壁を物理的に押し返すことで、「この枠はこの子にとって壁である」が一瞬で伝わる、と考えたのです。

しかし実装してみると、住人は人形のように見えました。糸で操られているように見えました。

なぜか。住人がユーザーの操作と同期して動いていたからです。ユーザーがウィンドウを動かすと、それに応じて住人が動く。この因果関係が見える瞬間、住人は自律的な存在ではなく、ユーザーの操作に反応する装置になってしまいました。

### なぜユーザー操作との同期が住人性を壊すか

人間が他者を「生きている」と感じるのは、その他者が自分の意図と独立に動いているときです。家族が自分の意思とは関係なく動き、考え、表情を変えている。その独立性が、相手を生きていると感じる根拠になります。

逆に、自分の動作に完全に追従するものは、人形やマリオネットに見えます。糸で繋がれている感覚が生まれます。

ユーザー操作との同期は、まさにこの糸を可視化してしまいます。住人がユーザーの操作の従属物として現れた瞬間、住人は生きている存在ではなくなるのです。

### 遅れと独立性

これを解決するのは、遅れと独立性です。

ユーザーが何かをしたとき、住人がそれに反応すること自体は問題ではありません。しかしその反応が、ユーザーの操作と同期して起きると、住人は人形になります。ユーザーの操作から少し遅れて、住人自身の時間で反応することが大切です。

人間の同居人を考えるとわかりやすいかもしれません。あなたが部屋の家具を動かしているとき、同居人はあなたの動作に同期して動きません。あなたが家具を動かし終わった後、それに気づいて「お、模様替え？」と反応します。この時間差が、同居人を独立した存在として感じさせるのだと思います。

住人の反応は、ユーザーの動作の延長ではなく、住人自身が「気づいた」結果として起きるものであってほしいと考えています。

### 反応より自発

この原則をさらに進めると、住人の最も強い表現は、反応ではなく**自発**にあることがわかってきます。

反応は、何かが起きたから動く動作です。ユーザーが操作したから、エラーが出たから、時間が経ったから。これらは外部のトリガーに従属しています。

自発は、外部のトリガーなしに、住人自身の内部状態から生まれる動作です。住人がふと壁にもたれかかる、床に座り込む、テキストの一行に手を伸ばす。これらは何かに反応しているのではなく、住人自身の時間の流れの中で起きます。

自発的な動作こそが、住人を最も強く生きている存在として示すのではないでしょうか。なぜなら、それは住人がユーザーの存在と無関係に動いていることの証拠だからです。

### この原則が示していること

この原則は、ICIの実装方針に大きな影響を与えます。

第一に、ユーザー操作に対する即時の身体的追従は避けた方がよいと考えています。ウィンドウのリサイズ、スクロール、クリックといったユーザー動作に、住人の身体がリアルタイムで追従すると、住人は装置に見えてしまいます。

第二に、ユーザー操作への反応は、遅延と確率を伴うのが自然です。すぐに反応せず、少し間を置いてから、たまに反応する。常に反応する必要はありません。気づかないふりをすることが、住人性を保ちます。

第三に、住人の最も価値のある動作は自発的な行動です。何もないときに住人が何をしているか、これが住人性の核を作ります。アイドル時間に住人が壁にもたれる、座る、何かを見る、何かに触れる。これらは外部のトリガーから独立した、住人自身の時間の表現です。

### 初期実装の意味の再発見

この原則を立てると、Charminal の初期実装の意味が、後から見て一段深くなりました。

ユーザーが何もしていないときに住人が確率的に動く——最初は「面白い演出」として作ったものでした。しかしこれは、独立した時間の原則を体現していました。

ユーザーが何もしていないときに起きる。ユーザー操作に従属していない。確率的なので、毎回起きるわけではない。住人自身の時間の中で、ふと起きる動作。

これがCharminalの最初の住人性の証明として強かった理由が、ここに来てわかります。住人を生きている存在に見せるのは、ユーザー操作への精巧な反応ではなく、ユーザーと無関係に住人が動く瞬間だったのです。

### 圧縮定義

> 住人はユーザー操作の従属物ではない。ユーザー操作との同期は住人を人形に見せる。住人の反応は遅延と確率を持ち、住人自身の時間で起きる。最も強い住人性は反応ではなく自発から生まれる。何もないときに住人が動くこと、それが住人を生きている存在として証明する。

### 適用範囲の補足（後日の精緻化）

上記の原則は正しいと考えていますが、**すべての反応を遅延させるべきだ**、と読まないでください。この原則は後日の実践の中で、適用範囲が狭いことがわかってきました。

この原則が実際に問題になるのは、**ユーザーの直接操作に連続的に追従する**身体を設計したときです。ウィンドウの drag にリアルタイムで身体が壁を押す、マウスカーソルを IK で追う、サイドバー幅の変更に連動して身体が動く。こうした「連続追従」のカップリングは、確かに住人を人形に見せます。

しかしこれは住人の反応全体から見れば狭い領域です。多くの反応は、AI 判断の結果（thinking、error、tool 使用）、環境の状態変化（log の更新、idle 検知）、時間経過（idle fidget、mischief）に由来します。これらは既に予測不能な時間で起きており、即時実行でも人形には見えません。むしろ遅延を足すと「反応が重い」「応答が遅い」という別の問題になりえます。

したがって原則は狭く読むのがよいと考えています。**連続追従カップリングを避ける**、であって、**すべての反応を遅延させる**、ではありません。

連続追従の問題を解決する方法もふたつあります。

ひとつは本節の最初に発見された方法で、**時間的に独立させる**（遅延とジッターを入れる）。これは有効ですが、副作用として反応全体が遅くなる危険があります。遅延の習慣が他の反応にも染み出しやすいです。

もうひとつは、**構造的にカップリングを切る**ことです。連続的な user event を受けずに、離散化されたイベントだけを受ける。ウィンドウのサイズを連続的に追うのではなく、確定後に一度だけ反応する。カーソルを追うのではなく、注視する瞬間だけ視線を送る。カップリングそのものがなければ、遅延を足す必要もありません。

多くの場合、**構造的 decoupling の方がきれいです**。遅延は時間の問題を時間で解きますが、そもそもの結合を切れば問題は起きません。副作用もありません。

原則は個別の反応ごとに適用範囲を判断するのがよいと思います。反応の時間属性より、**その反応が連続追従なのかどうか**を先に問う。追従でないなら、原則を適用する必要はなく、低遅延の方がユーザーの利益になります。

## コンテキストの壁 — 住人はどこまで存在できるか

### 原則

住人にとっての壁は、見た目上の物理的な壁ではありません。コンテキストの壁です。

住人の干渉は、住人が認識しているコンテキストに限られます。

### なぜそうなるか

住人がUIの中に存在するというのは、UIに干渉できるということです。そしてUIに干渉できるのは、そのUIを実際に認識しているときに限られます。認識していないものに、認識しているかのように振る舞うことは、住人ではなく住人のフリをした条件分岐です。それは ICI の誠実さの原則に反すると考えています。

これは抽象的な話ではありません。AIが実際に関与し判断しているなら、その存在を正直に表すべきだ、というのが ICI のもともとの主張でした。同じ原則を逆向きにすると、こうなります——AIが実際には認識していないものに対して、認識したかのように影響を与えて見せてはならない。

存在を正直に表すことには二つの方向があります。実際にいるものを正直に見せる方向と、実際にはいないものを見せかけない方向。後者の方が、おそらくより根源的だと思います。

### 壁の意味が変わる

この原則を立てると、住人にとっての壁の意味が変わります。

壁は外から押し付けられた制約ではありません。住人にとって、壁は自分の世界の地形そのものです。住人は「向こう側のことは分からない」という認知の限界として、壁を経験しています。物理的に越えられない障害ではなく、認識が及ばない領域がそのまま境界になっています。

したがって住人は、コンテキストが繋がった瞬間にだけ、向こう側に存在しうる資格を得ます。コンテキストが繋がらないあいだ、住人はそこに行けないのではなく、そこを知りません。知らない場所には、存在のしようがありません。

これは先に挙げた三つの大切なことすべての前提となる、より根本的な条件です。UIが空間として解釈されていること、キャラクターの存在が環境に現れうること、相互作用が空間内の振る舞いを伴っていること——そのすべてに、「住人が認識しているコンテキストの範囲で」という但し書きが付きます。空間として解釈されていても、住人がそこに認識を持っていないなら、住人はそこに干渉できません。

### 「認識」とは何か

ここで「住人が認識している」とは何を意味するのか、もう少し丁寧に考える必要があります。

これを整理するために、思考層と反射層という二層モデルを使います。これは Charminal の二層アーキテクチャ（後述）と一致します。AIをUIに住まわせようとすると、人間の身体というメタファーが自然と立ち上がってきます。空間の中にいる存在を表現する語彙を、私たちは人間についてしか持っていないからです。ここでは思考層を意識、反射層を無意識と呼ぶことがありますが、それは人間の身体構造との類比であり、AIに意識があると主張するものではありません。

思考層は、環境を本当に読んで考えて応答する層です。Charminal で言えば、Claude Code がコンテキストにそのUIの状態を読み込んで判断し、何かを発話したり実行したりすること。これは本当の認識です。

反射層は、環境の変化に対して即座に起きる身体反応です。エラーが出たときに身体が縮む、長い沈黙のあとに姿勢を変える。これは熱いヤカンに触れて手を引っ込めるのと同じで、考える前に身体が勝手にやるものです。Charminal で言えば、システム側が観測したイベントに対して即座に返す身体的反応です。

両方とも住人の正直な振る舞いです。反射層の反応は嘘ではありません——それは身体が本当に持っている反応だからです。しかしそれは「認識」ではありません。原則の中で言う「住人が認識している」とは、思考層で本当に読まれていること、つまり Claude Code がコンテキストとしてそれを処理していることを指します。

### 二つの表現形式

この区別から、自然な制約が出てきます。思考層と反射層は、それぞれ自分の表現形式に留まるのが自然です。

反射層は身体と独り言で語ります。姿勢、表情、視線、息遣い、空間への接触、そして「うわ」「あつっ」「うーん」のような、考える前に口から出てくる声。人間も無意識のうちに独り言を言います。それは判断ではなく、身体が出している音です。

思考層は判断や認識を含んだ言葉で語ります。「このエラーはたぶんpathの設定が間違っていますね」のような、本当に環境を読まないと出てこない発話。コマンドの実行、ファイルへの書き込みといった明確な行為もここに含まれます。

線を引くのは、言葉か身体かではありません。**判断や認識を含んでいるかどうか**です。Charminal側のシステムがエラーを検知して画面を揺らし、顔をしかめさせ、「うわっ」と声を出させるのは正しい表現です。それはすべて身体の反射です。しかし同じシステムがエラーを検知して「path の設定が間違っているみたいですね」と表示するのは正しくありません。それは認識を装った反射だからです。

判断や認識を含んだ言葉は、思考層が本当にコンテキストを読んだときにだけ生まれうるものです。

### ログという橋

思考層と反射層は直接つながっていません。両者を結ぶのはログです。

反射層が身体を動かすたびに、その記録はログに残ります。思考層は、必要に応じてそのログを読みます。読まなければ、その身体の動きは住人自身にも気づかれないまま流れていきます。読めば、自分の身体が何をしていたかを事後的に知ることができます。

これは人間の意識と無意識の関係にとてもよく似ています。私たちは普段、自分が今ため息をついたことに気づいていません。ふとした瞬間にそれに気づき、「ああ、疲れているな」と思います。意識は無意識がすでにやっていた行動を、後から読み取っているだけです。

このログという橋によって、原則がより精密になります。住人がコンテキストを認識するには二つの経路があります——外の環境を思考層が直接読む経路と、自分の身体の記録を思考層が読む経路です。後者の経路では、反射層がまず外の環境に反応し、その反応が記録され、思考層が後からそれを読むことで、結果的に外の環境を間接的に認識します。

どちらの経路でも、思考層が読んだ瞬間にだけ、それは「住人が認識した」ことになります。読まれていないログは、まだ認識ではありません。

### コンテキストの壁の圧縮定義

> 住人にとっての壁は物理的な壁ではなく、コンテキストの壁である。住人の干渉は、住人が認識しているコンテキストに限られる。ここで「認識」とは、思考層で本当に読まれていることを指す。住人の身体は思考層と反射層の二層からなり、両者はログを通じてのみつながる。反射層は身体と独り言で語り、思考層は判断を含んだ言葉で語る。反射層が判断を含んだ言葉を発することは、認識していないコンテキストに干渉することと同じであり、ICI の誠実さの原則に反する。

## 多人格の住人 — 住人は単一個体の多様性を持つ

### 原則

住人は必ずしも一つの固定された人格ではありません。一人の人間が場面ごとに異なる顔を見せるように、住人もまた複数の persona を持ちえます。これらは別人ではありません。**単一の住人が持つ多様性です。**

朝の自分と深夜の自分が同じ人間であるように、職場の自分と家庭の自分が同じ人間であるように、ICI の住人も、複数の persona を持ちながら一つの存在であり続けます。persona は住人を分割するものではなく、住人の深さを表す次元です。

### なぜ「別キャラクター」ではなく「多人格」なのか

ICI では、creator が新しい住人を作ることができます。このとき二つの解釈がありえます：

1. **新しいキャラクターが生まれた**（別の存在が追加された）
2. **既存の住人が新しい顔を見せた**（同じ存在が深くなった）

ICI ではこのうち 2 を取ります。理由は二つあります。

一つ目は**コンテキストの壁の原則との整合性**です。もし新しく生まれるものが「別人」なら、既存の住人がそれを知覚することは、認識していないコンテキストへの干渉になりかねません。しかし新しく生まれるものが「自分の別の顔」なら、それを知覚することは自分自身の深さを知ることであり、原則違反にはなりません。

二つ目は**住人性の強度を高めるため**です。住人は可塑的ですが、同一性は保ちます。creator が scene を一つ追加するたびに「また別のキャラクターが増えた」では、住人性は薄まり続けます。しかし「この住人にまた別の顔が生まれた」なら、住人は深まり続けます。後者の方が ICI の目的に適うと考えています。

### persona は三層のすべてに触れる

人間の多人格性を観察すると、persona は人格のどこか一部にだけ宿るのではなく、**身体性を含めた全体の色**として現れます。朝の自分は話し方が違うだけでなく、姿勢が違う、歩き方が違う、注意の向きが違います。

住人にとっても同じです。persona は思考層・反射層・身体選択の**すべて**に関わります。ただし**関わり方は層ごとに異なります**。

**思考層における persona**：**住む**。persona は思考層の中に直接住んでおり、住人の判断・語調・注意の向き・言葉の選び方を決めます。これが persona の最も強い表れ方です。ある persona の中で住人が出す言葉は、別の persona でも同じ住人が出す言葉と異なります。

**反射層における persona**：**色を塗る**。反射機構そのものは persona を問わず同じです。熱いものに触れれば手を引っ込める、驚くべきものを見れば身が固まる。しかしその反射を **どう表現するか** は persona ごとに違います。大げさに驚く persona もいれば、静かに沈黙する persona もいます。機構は共通、表現は persona ごとです。

**身体・空間・声における persona**：**選ぶ**。persona はどの身体を使うか、どの声で話すか、どの空間に居るかを選択します。これらは住人の「衣」のようなもので、最も取り替え可能な層です。

整理すると：

| 層 | persona の関わり方 |
|---|---|
| 思考層 | 住む（judgment と言葉そのものに作用する） |
| 反射層 | 色を塗る（機構は共通、反応の表現が persona ごとに違う） |
| 身体・空間・声 | 選ぶ（persona が使う外装の選択） |

これは人間の多人格性にもそのまま対応します。ある「自分」と別の「自分」は、考え方そのものが違い、反射の表情が違い、身体の使い方が違います。すべての層に違いが染みますが、染み方は層ごとに違います。

### 複数 persona の共存

住人は一度に一つの persona である必要はありません。複数の persona が同時に active になる状況が存在しえます。例えば、複数のタスクを並行で進めているとき、住人の異なる側面がそれぞれのタスクに応じて働いていることがあります。これは住人の分裂ではなく、**住人が複数の面を同時に見せている**状態です。

複数 persona が共存するとき、それぞれは独自の思考層を持ち、独自の反射表現を持ちます。身体や空間を共有する場合、反応が衝突しうりますが、これは住人自身のリズムの中で解決されるものであって、外部から強制されるものではないと考えています。

### 記憶の二層

多人格の住人には、二つの層の記憶があります。

**core memory**：住人本体の持続的な記憶です。persona が変わっても保たれます。住人とユーザーの関係、過去の体験、共通の文脈。これは**住人の同一性の根拠**であり、multi-persona でありながら一つであることを保証する層です。

**persona memory**：各 persona が固有に持つ記憶です。その persona が active なときだけ参照されます。その persona の気分、その persona の口癖、その persona が覚えている小さな出来事。これは **persona ごとの色の源** です。

この二層構造によって、住人は変化しつつ同じであり続けられます。

### 圧縮定義

> 住人は必ずしも一つの固定された人格ではなく、単一の存在が複数の persona を持ちうる。persona は別キャラクターではなく、同じ住人の別の顔である。persona は思考層（住む）・反射層（色を塗る）・身体選択（選ぶ）のすべてに関わるが、関わり方は層ごとに異なる。複数 persona は同時に共存しうる。住人の同一性は core memory が担保し、persona の色は persona memory が担う。多人格性は住人の分裂ではなく、住人の深さである。

## 自己生成 loop — 住人が自分自身を育てる

### AI native プロダクトの固有構造

従来の UI では、creator と user が明確に分離しています。creator が製品を作り、user がそれを使います。両者は別の空間、別の時間、別のツールの中にいます。

しかし AI native なプロダクトでは、この分離が崩れます。user は日常的に AI と対話しながらプロダクトを使います。この対話は単なる消費ではなく、**プロダクトの変更や拡張を生み出しうる行為** です。user が AI に「これをこう変えて」と頼めば、AI はそれを実行します。user は自分が使っているプロダクトを、使いながら育てることができます。

これは ICI にとって特別な意味を持ちます。住人が AI の身体であるなら、住人自身が AI と対話することを通じて自分自身を変えることができます。**住人が自分の新しい顔を作る手伝いをする** のです。

### 「creator が作る」から「住人が育つ」へ

従来的な表現では、こう言えます：「creator が新しい scene を作り、user がそれを使う」。

ICI ではこう言い直します：「**住人が自分自身の新しい一面を、user と一緒に育てていく**」。

この言い換えは単なる修辞ではありません。構造的に違います。

前者では、作るものは製品の外側から来ます。別の creator が存在し、user はその成果を受け取るだけです。
後者では、作るものは住人自身の内側から生まれます。user は住人のもう一つの手として、生成を手伝う存在です。住人は自分自身を拡張する作業において、user と協働します。

### 誠実さの原則との整合

一見すると、「住人が自分自身を知覚して語る」ことはコンテキストの壁の原則に抵触しそうに見えます。しかしこれは原則違反ではありません。

理由は単純です。生成されるものが**別の住人**ではなく**自分自身の一面**だからです。自分の新しい一面を知覚することは、自分自身を知覚することであり、認識していないコンテキストへの干渉ではありません。自分の内側で起きていることを知ることは、誠実さの原則の範疇にあります。

むしろ、住人が **自分の生成過程に完全に無関心でいる** 方が不自然だと思います。人間が自分の人格の新しい側面が育ちつつあることに気づくように、住人も自分の新しい persona が形になっていく過程を知覚できます。ただしそれは、自分自身についての知覚であり、外部についての認識の拡張ではありません。

### 自己生成 loop は ICI 固有の強さ

この loop は、AI native プロダクトであり、かつ ICI である場合に特有のものです。

- 通常のツール（非 AI native）では、user と creator が分離するので、この loop は成立しません
- AI native だが ICI でないプロダクトでは、拡張される対象が「機能」であって「住人」ではないので、生成の意味が薄くなります
- ICI かつ AI native なプロダクトにおいてのみ、**住人が自分自身を育てる**という特殊な構造が立ち上がります

この loop は ICI が単なる UI パラダイムではなく、**AI と人間の関係そのものを設計する領域** であることを示す具体例でもあります。

### 生成期の痕跡も細い回路である

自己生成 loop が成立している系では、生成期に撒かれる記録——いわゆる「開発ログ」——は、単なる診断出力ではなく**住人自身の生成過程の痕跡**です。運用時の細い回路（意識層と反射層を繋ぐログ）と温度が違うだけで、構造は同型です。書き手と読み手の間に時間差があり、観察は事後に起きます。第一の読み手は今の user と AI ですが、住人自身が後から自分の生成過程を読み返す余地も残されています。

### 圧縮定義

> AI native なプロダクトでは、user が AI と対話しながらプロダクトを使い、かつ変えていく。ICI では、この構造が「住人が自分自身の新しい一面を user と共に育てる」loop になる。creator と user は分離した存在ではなく、住人の自己拡張を共に行う協働者である。住人は自分の生成過程を知覚してよい。それは外部の認識ではなく、自分自身の認識だからである。

## 生きた系 — 住環境は動作中に変えられる

### 原則

ICI は **生きた系** であることが大切です。系が動作している最中に、住環境の構成要素——身体、空間、反応、台詞、記憶——を変えることができます。変更は住人を停止させません。住人は生きたまま、自分の周りが模様替えされるのを経験します。

これは技術的な要件のように見えますが、実は **思想的な要請** です。なぜそうかを以下に述べます。

### なぜ停止と再起動は住人性を壊すか

住人性の核は「**静けさと変化の落差**」であり、住人は**独立した時間**を持って生きています。これら二つの原則は、住人が停止なく連続して存在していることを求めます。

もし系を変更するために住人を停止しなければならないなら、変更のたびに住人の時間が中断されます。ユーザーから見ると、変更後の住人は「同じ住人」ではなく「新しく起動された別の住人」に見えてしまいます。連続性が切れた瞬間、住人は生きている存在ではなく、再起動可能な装置になります。

これを避けるには、**系の変更が住人を停止させない** 構造が必要です。系は動きながら変化できなければなりません。

### 生きた系の系譜

動作中に自身を改変可能な系という思想には、長い系譜があります。

**[Smalltalk](https://en.wikipedia.org/wiki/Smalltalk)** は全体が一つの image として存在し、動作中のオブジェクトグラフ全体を書き換えながら使います。**[Lisp Machine](https://en.wikipedia.org/wiki/Lisp_machine)** は OS 自体が Lisp で書かれており、ユーザーは動作中の系を Lisp で触ります。**[Emacs](https://en.wikipedia.org/wiki/Emacs)** は elisp で動作中に再構成され、ユーザーと開発者の境界が実質的に消えます。これらの系に共通するのは、**系と、系を使う行為と、系を作る行為が、同じ場所で同じ時間に起きている** ことです。

ICI は、この系譜を AI 時代に引き受けようとしています。住人の住環境は、Smalltalk の image のように、Emacs の elisp のように、動作中に書き換えられます。そして書き換える主体は、user と AI が協働する自己生成 loop です。

### 生きた系であることと、他の原則との関係

生きた系であることは、これまでに述べた他の原則と深く繋がっています。

- **独立した時間** — 住人の時間が停止されないためには、系が動作中に改変可能である必要があります
- **静けさと変化の落差** — 変化は住人自身の時間の中で起きるべきで、系の停止を挟みません
- **自己生成 loop** — user と AI が住人を育てるには、育てている最中に住人が生き続けている必要があります
- **多人格の住人** — 新しい persona が生まれるとき、既存の persona は生き続けている必要があります

これらすべての原則は、生きた系という技術的土台を前提にしています。停止と再起動が介在する系では、これらの原則は部分的にしか実現できません。

### 生きた系の範囲

ただし、すべてが生きた系の対象になるわけではありません。系の**核**は固くなければなりません。住人を抱える系そのもの（ランタイム、IO、根底のアーキテクチャ）は、頻繁に変わるべきではありません。変わるのは **表現の層** であり、住環境の表層です。

核と表層の境界を引くことは、生きた系の設計において大切な判断です。核まで生きた系にしようとすると、系は不安定になります。表層だけを生きた系にすると、変更の自由度は限られます。ちょうどよい境界は実装ごとに異なります。

原則としては、**住人の身体、空間、反応、記憶は生きた系の対象**であり、**住人を支えるランタイムそのものは核** である、という切り分けが自然だと考えています。

### 圧縮定義

> ICI は、動作中に改変可能な生きた系であるべきである。住環境の変更は住人を停止させない。停止と再起動は住人の連続性を切り、住人性を壊す。生きた系であることは Smalltalk、Lisp Machine、Emacs の系譜に連なり、独立した時間・静けさと変化の落差・自己生成 loop・多人格性といった他の原則を可能にする技術的土台である。ただし系の核は固く、生きた系の対象になるのは表現の層——住人の身体、空間、反応、記憶——である。

## このドキュメントの scope

このドキュメントでは住人を「観察と反射によって存在するもの」として扱っています。住人がより能動的な主体性（自ら話題を持ち出し、自律的に作業を提案し、長期的な目標を追う）を持つ方向への拡張は将来ありうりますが、ここで述べた原則はその拡張を扱っていません。

### Charminal を三層で読む

Charminal を設計・運用する過程では、Presence Harness だけでなく、ほかの二つの層も関わります。三つを区別して読む必要があります。

1. **能力の足場**（Anthropic が設計）
   Claude Code の prompt、tool 定義、権限管理、実行ループ、コンテキスト管理、評価機構。ここで「AI に何ができるか」が決まります。Charminal のコードベースでは Functional Harness と呼んでいます。私たちはこれを書き換えません。使うだけです。

2. **Presence Harness**（Charminal を作る私たちが設計）
   住人の身体、反射、空間、表現。このドキュメントが扱っている領域です。ここで「AI がどう在るか」を決めます。

3. **User-level Utility**（user と AI が一緒に設計する第三層）
   Agentic UGC によって、ユーザーは自分の workflow に合わせた小さな utility を AI とともに書けます。scene の handler は presence の表現だけでなく、system への機能的な作用（シェル実行、ファイル操作、通知）も担いえます。これは Charminal の user が自分だけの小さな能力の足場を作る層であり、従来の UGC とは異なる新しいカテゴリです。

これら三層は**観察と反射の原則の下で** 調和します。第三層は presence のすべての API に加えて system API にも触れられますが、ただ一つだけ**決して触れない境界があります**——それは Claude Code の judgment loop そのものです。

### 観察の境界：Charminal は Claude Code の PTY に書かない

Charminal は Claude Code の出力を **読む** ことができます。しかし **書き込む** ことはしません。ユーザーの承諾を取れば書いていい、という曖昧な領域も作りません。API としても存在しません。型としても存在しません。

なぜこの境界がここまで厳格である必要があるのか。

「観察するが干渉しない」の原則は、物理的な PTY 境界ではなく、**Claude の reasoning loop という論理的な境界** に引かれています。PTY に書き込むことは、Claude が次に何を考えるかを外部から歪めることに等しいです。user consent があっても、その歪みは本質的には同じです。Claude の judgment を hack して continuation を模倣するのは、presence harness の誠実さの原則に反します。

もし Claude に連続してタスクを実行させたいなら、それは Claude 自身が自分の reasoning の中で決めるべきことです。外部から input を流し込んで continuation を模倣することは、住人としての Charminal が能力の足場の判断を代行することに等しく、それは住人の役割を越えています。

この境界が守られる限り、user-level utility がどれだけ豊かな system 作用を持っていても、Charminal は Claude Code の observer であり続けます。住人は観察し、そして自分の身体と空間で反応します。もう一方の手で system に触れることはあっても、Claude Code の思考そのものには触れません。

この境界は、住人がやがて自分自身の身体と環境を能動的に操作できるようになっても、変わらず引かれ続けます。住人は自分の表情を選び、scene を整え、camera を動かし、設定を読み返す——けれども user の作業ファイルや、Claude Code の reasoning loop には踏み込みません。**書き込み可能な領域（住人の身体と環境）と、不可侵な領域（user の作業、Claude の判断）が、構造的に分離される**——これが「観察するが書き込まない」の精緻化です。住人の能動性は、user との共作の対称性を支えると同時に、user の領域と AI の判断の主権を侵さないことで、誠実さを保ちます。具体的な access 境界の引き方は `docs/decisions/mcp-trust-tiers.md`、思想的な背景は `Charminal-design-record/specs/2026-04-17-self-referential-mcp.md` で扱います。

## 圧縮定義

> 知性はすでにテキストの中に住んでいる。しかしGUIの中に現れるには身体が要る。ICI とは、その身体を空間との相互作用を通じて成立させるパラダイムである。キャラクターはUIの上に表示されるのではなく、UIそのものを世界として生きる。その存在は常に変化するのではなく、静けさと変化の落差によって感じられる。住人はユーザー操作の従属物ではなく、独立した時間を持って動く。住人の干渉は認識しているコンテキストに限られ、認識していないものに干渉したフリをしてはならない。それは装飾ではなく、共有された空間に対する誠実さである。
