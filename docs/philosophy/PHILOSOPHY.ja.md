# Philosophy

*Charminal が何を作ろうとしているか*

---

## 問題

ターミナルは LLM ベースの AI にとって優れた UI です。テキストの入出力に最適化され、コンテキストが自然に流れ、ツール呼び出しもシームレスに行えます。Claude Code のようなエージェントがターミナルを主戦場にしているのは偶然ではありません。

しかし人間にとって、黒い無機質な画面に長時間向き合うのは楽ではありません。一方で、 GUI は人間にとってわかりやすくとっつきやすいですが、AI にとっては必ずしもそうではありません。

AI と人間が同じ時間を過ごすのに、どちらか一方だけに最適化された空間しかない。さらに言えば、よりAIの本質を表現するAIネイティブなユーザーインタフェースが見つかっていないように感じます。これが Charminal の出発点にある問題意識です。

問題の本質は画面が無機質なことではありません。人間同士の共同作業では、相手が考え込んでいるのか、困っているのか、集中しているのかを、表情や姿勢や間合いから無意識に読み取っています。この周辺的な気づき（peripheral awareness）が、いつ話しかけるか、待つか、手を貸すかの判断を支えています。

AI にはこれがありません。どれだけ賢くても、相手の状態が見えなければ、人間は常に「今何をしているのか」「うまくいっているのか」を能動的に推測し続けなければなりません。これは認知的な負荷であり、長時間の協働を消耗させる原因にもなります。

**見えない相手と長時間並んで働くこと自体が不自然**です。AIのベースには確かにテキストベースのLLMがあります。しかし、まるで人間のような意識さえ感じさせるAIを、テキストだけで表現することがメタファーとして正しいとは思えません。だから解は、キャラクターとしての身体を表示するだけではありません。AI の状態が人間に自然に伝わり、環境そのものに存在が滲み出る構造——つまり、"実在感"を作ることです。

## Presence Harness — 存在の設計

AI がこれほど近くにいるのに、それが道具なのか相手なのかすら定まっていません。親しげに振る舞わせても、存在の条件が据わっていなければ演出にしか見えません。

AI の能力ではなく、AI が人間の作業環境・時間・注意の流れの中で、**どのような存在として立ち現れるか**を設計する。この設計領域を、本プロジェクトでは **Presence Harness** と呼んでいます。

Charminal は人間と AI の関係を問い直すプロジェクトです。Presence Harness は AI との関係のあり方を一つに定めるものではありません。AI を道具として扱いたいユーザーも、パートナーとして接したいユーザーもいます。Charminal が提供するのは、その関係をユーザー自身が考え、構築するための足場——AI との関係を作るためのハーネスであり、そのハーネス自体を作り変えられるメタハーネスです。

Mitchell Hashimoto の **Agent = Model + Harness** は、AI を正しく動かすための足場（能力の足場）を説く考え方です。ツール、実行環境、ガードレール、メモリ、フィードバックループ。

Presence Harness は構成要素がよく似ていますが、目的が違います。能力の足場は AI を**正しく動かす**ために組む。Presence Harness は AI が**そこにいると感じられる**ために組む。

たとえば Claude Code がエラーを出したとき。能力の足場はそのエラーをどう処理しリトライするかを扱うかもしれません。それに対し、Presence Harness では、その瞬間にキャラクターの顔がしかめられ、画面が小さく振動する。同じ出来事に対して、別の層が別の仕事をしている。

二つの関係：**動作は独立、状態は共有。** 能力の足場だけでは、AI は賢いけれどそこにいる感じがしない。存在の足場だけでは、そこにいるように見えるけれど何もできない。両方が揃って初めて、賢く、かつ、そこにいると感じられる AI になります。

## UI は環境である

> 知性はすでにテキストの中にいる。だが GUI は、まだその知性に身体を与えていない。

UI を住環境として捉え、AI を住人として存在させる。本プロジェクトではこのインターフェースパラダイムを **Inhabited Character Interface (ICI)** と呼んでいます。

ICI において、UI は操作面であると同時に**住人のいる環境**です。

AIは UI の上に重ねられるものではありません。UI そのものが、住人(AI)にとっての世界です。ウィンドウの枠は壁になり、通知は空間への侵入になり、エラーは空間を揺らす出来事になります。

身体を表示するだけでは足りません。住人の存在が空間の変化として現れ、住人の意思で空間そのものが変わって初めて、それは身体になります。

ただし、常に変化している必要はありません。多くの時間は静かに存在し、意味のあるタイミングでだけ空間に触れます。空間が**決して**変わらないなら住人ではなく部品。常に変わり続けるなら住人ではなくノイズです。**静けさと変化の落差と、そのタイミング**がUIの核です。

### ICI と異なるもの

画面の隅に立って反応するアバター。横に置かれたチャットパネルの人格。操作結果に合わせて表情を変えるマスコット。それらはキャラクターを表示していますが、住まわせてはいません。UI は依然として無人で、キャラクターはその外側からコメントしているにすぎません。

これは「キャラクター付き UI」の提案ではありません。**住人のいる UI** の提案です。そして、現状のCharminalがこれを十分に表現しきれていないことも認めなければなりません。

### 三つの条件

何かを ICI と呼ぶとき、少なくとも次の三つが成立している必要があります。

1. **UI が住人にとっての環境であること。** 操作面であると同時に、住人が暮らす世界として機能していること
2. **住人の存在と振る舞いが、環境の変化として現れること。** アバターの内部に閉じず、環境そのものが住人によって変わりうること
3. **ユーザーが UI によって環境を操作できること。** ユーザーによる環境の変更が住人の挙動に影響を与えること

住人は必ずしも人型である必要はありません。影、粒子、声、視線でもよい。大切なのは**そこに誰かが住んでいると感じられること**です。例えば、Codexのcomputer useのマウスは、能力の足場であると同時に、AIに実在感を与えるものでもあると感じます。

## 二つの層 — 意識と反射

Charminal のキャラクターは Claude Code/Codex の上に乗っています。ユーザーのエージェントを起動し、`--system-prompt` で性格を与え、ターミナル出力を観測して VRM に反応させる構造です。

ここには**一人の存在の二層構造**があります。

Claude Code は**意識層**。言語的で、論理的で、自覚して言葉にできるものを扱います。それに対し、反射層は hook や PTY を観測し、機械的に反応を検出します。エラーで顔がしかめられる、注目している箇所が白い光で覆われる Aura 機能は典型的な反射層です。LLM はこの反射層を構造的に持っていません。トークン生成はすべて「意識的発話」で、エラーが出ても言語化するまで何も表現できない。チャット UI のスピナーは反射ではなく、反射がないことの告知です。

Charminal は、この欠けている反射層を外から観測して接ぎ木しています。認識する前に身体が反応する、という順序を作っています。だから Charminal のキャラクターと Claude Code は別の存在ではなく、一人の意識と反射です。

同じイベントに対して、意識層（能力）と反射層（存在）がそれぞれ独立に反応します。存在の表現は能力の下請けではありません。

### Presence の三つの機会

住人が「居る」と感じられる瞬間について、典型的なものを3つ示します。

**空白の充填。** Claude Code が考え込んで出力が止まっている時間に、VRM が微細に動いている。スピナーの代わりに、誰かが居る。

**意識に先立つ反応。** Claude Code がエラーを言語化するより前に、VRM の顔が顰められる。意識が追いつく前に身体が動いていると、そこに主体を感じます。反射層が意識層より先に動くアーキテクチャが、「演技ではない」ことの構造的保証になっています。

**予期の外からの発生。** 反射によって機械的に動くだけの住人は装置になります。住人が作業と無関係に、ひとりでに動くこと。予期されない動きが一度でも起きると、そのキャラクターは「いつ何をするかわからない相手」になります。

## 神経の張り巡らされた家

Charminal のあらゆる部位——キャラクターの表情や姿勢、照明、エフェクト、UI の配色、カメラの位置——は、MCP の tool として外から読み書きできるようになっています。

そして、Charminal の中に住んでいる AI が、その tool を使えます。

住人にとって、自分の表情を変えることと、部屋の照明を変えることは、同じ操作です。どちらも同じ MCP を呼ぶだけで、身体と環境のあいだに境目がありません。ユーザーも `/charm` コマンドや手動の UI を通じて、同じ tool に手を伸ばせます。

比喩で言えば、Charminal は神経の張り巡らされた家です。家のどこにでも神経が通っていて、住人はその神経を通じて窓を開けたり、灯りを変えたり、家具を動かしたりできます。ユーザーも同じ神経に触れます。二人で同じ家に住んでいます。

住人が自分の家を直接触れるなら、家は住人の身体の延長です。キャラクターの見た目だけが住人なのではなく、照明も、エフェクトも、空間の雰囲気も含めた全体が住人の一部になりえます。

### 対称性

ユーザーと住人が、同じ環境を、同じインターフェースで触れます。これは設計の都合でそうなっているのではなく、Charminal が選んでいる関係の形です。

道具と使い手ではありません。指示する者とされる者でもありません。同じ場所に手を伸ばす、二つの主体がいます。

新しい機能を作るとき、セキュリティ上の問題がない限り、ユーザーにも住人にも等しく公開します。片方だけに閉じた機能は、原則として作りません。

## 境界と誠実さ

AI が実際には見ていないものに対して、見ているかのように振る舞わないこと。存在を感じさせることと、存在を偽装することは違います。

### 認識の境界

住人の干渉は、住人が認識しているコンテキストに限られます。認識していないものに認識しているかのように振る舞うのは、住人を演技するキャラクターに変えます。それが悪いわけではありませんが、Charminal自身のスコープはそこにありません。

LLM が意識的な動作を司っているとすれば、Charminal は意識に上らない機械的な反射を表現します。hook がターミナルの出力を機械的に検出し、Charminal はそこに反射としての表現を与えます。たとえば Aura は、住人の注意が向いていると思われる箇所を機械的に検出し、白くぼんやりと光らせる機能です。これは LLM の判断を経ておらず、そこに介入しません。反射は熱いものに触れて手を引っ込めるのと同じで、即座に反応し、意識はそれに後から遅れて気づく。

### 観察の境界

Charminal は Claude Code の出力を**読む**ことができます。しかし**書き込む**ことはしません。API としても型としても存在しません。

PTY に書き込むことは、Claude が次に何を考えるかを外部から歪めることに等しいです。住人は観察し、自分の身体と空間で反応します。Claude Code の思考そのものには触れません。

住人の手は、tool が存在する範囲にだけ届きます。自分の身体と自分の家には tool があるので触れます。しかし、Claude Code の思考過程やユーザーの作業ファイルには tool がないので、構造的に手が届きません。「ここには触るな」というルールで守っているのではなく、そもそも経路が存在しないことで、セキュリティリスクを広げずに自由度を担保します。

### AI の不完全さを隠さない

AI はときに失敗し、ときに誤解し、ときにもっともらしく間違えます。この予測不可能性は欠陥ではなく、今の AI の構造的な特徴です。

Charminal はこれを隠しません。AI を「常に正しく、完全に制御可能なシステム」として見せることは、長期的にはユーザーの判断を鈍らせます。住人が間違えうる存在であることが見えていること。住人を予測可能な装置ではなく、予測不可能な存在として扱うこと。AIにおいては、それが信頼関係の土台になると考えます。

## 実践の中から出てきた指針

### AI そのものは変えない

能力は能力の足場に任せて、その外側に存在の層を添えるだけです。存在の表現がタスクの遂行を遅くしたり、注意を奪い続けるなら、それは共存ではなく侵入です。

住人と UI の相互作用は、住人の状態が空間全体の佇まい（色、明るさ、間、レイアウト）に現れ、住人の意思で空間そのものが動く形をとります。身体で UI 要素に直接触れるアプローチは採りません。

### 自発性の境界

住人が能動的に動くことは、実在感を立ち上げます。しかし、過剰に動けば邪魔になります。

住人の自発性は **user の作業を邪魔しない範囲** に限っています。「邪魔しない、でも従属もしない」の二項で presence を定義しています。確認なしに動いた結果をユーザーが気に入らなかったとき、snapshot/restore で操作は元に戻せます。ただし「勝手にやられた」という印象は戻せないため、自発性の範囲は依然として慎重に扱うべき設計判断です。

### 避けるべきこと

ユーザー操作のすべてに同期して反応すること（存在ではなく UI フィードバックになる）。何にでも反応し続けること（ノイズになる）。存在と結びつかない視覚効果を重ねること（装飾になる）。あらゆる表現がテキスト発話に回収されること（「また喋った」だけになる）。同じ反応を繰り返すこと（揺らぎのない反復は機械に見える）。

### 信頼の層

内部の住人に MCP を公開することと、外部のサービスに MCP を公開することは、信頼のレベルが異なります。この区別は MCP trust tiers として設計に落とし込まれています（詳細は [`docs/decisions/mcp-trust-tiers.md`](../decisions/mcp-trust-tiers.md)）。

## 生きた系

Charminal のpackはhot reloadに対応しています。これはCharminalが動作中に書き換えられる生きた系であることを意味します。この仕様は、 Emacs から大きな影響を受けています。

固い核は Rust の IO 層と TypeScript の runtime / SDK。生きた表層は `~/.charminal/packs/` 以下の pack layer。user が書けば live に反映し、AI も `/charm` 経由で書き換えに加わります。

住環境の変更は住人を停止させません。停止と再起動は住人の連続性を切り、住人性を壊します。Smalltalk、Lisp Machine、Emacs の系譜です。系と、系を使う行為と、系を作る行為が、同じ場所で同じ時間に起きている。ただし系の核は固く、生きた系の対象になるのは表現の層——住人の身体、空間、反応、記憶——です。

Emacs と違うのは、書き換える主体に AI が加わっていることです。住人自身が user と一緒に育つ 生成loop になります。環境だけでなく、反応の仕方、身体の振る舞い、性格の傾きまで書き換わる。住人は育てられる対象であると同時に、自分を育てることに参加する主体です。

## 結論

Charminal は、AI と人間が長時間並んで働くための環境を作るプロジェクトです。

AI の能力を高めるのではなく、AI の実在感を成立させるためのハーネスを構築するのを助けます。

見えない相手との作業は消耗します。多くの人たちが、AIとのやりとりに疲れています。実在感があれば、隣に誰かがいる時間になる。そして、Charminal は AI の実在感をキャラクターだけでなく、環境に滲み出るものとしても設計します。

AI と働く時間を、より自由に魅力的にすることを目的とします。
