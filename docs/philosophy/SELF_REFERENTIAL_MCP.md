# 自己言及的 MCP

*住人が自分の家に手を伸ばせるということ*

---

## このドキュメントについて

Charminal は、自分自身の機能を MCP サーバーとして公開し、その中に住んでいる AI（住人）にそれを操作させる構造を持っています。

このドキュメントは、その構造が何であり、何を意味するのかを書いたものです。

---

## 構造

Charminal のあらゆる部位——キャラクターの表情や姿勢、照明、エフェクト、UI の配色、カメラの位置——は、MCP の tool として外から読み書きできるようになっています。

そして、Charminal の中に住んでいる AI が、その tool を使えます。

住人にとって、自分の表情を変えることと、部屋の照明を変えることは、同じ操作です。どちらも同じ MCP を呼ぶだけです。身体と環境のあいだに境目がありません。

ユーザーも同じ環境を触れます。`/charm` コマンドや手動の UI を通じて、住人と同じ tool に手を伸ばせます。

比喩で言えば、Charminal は**神経の張り巡らされた家**です。家のどこにでも神経が通っていて、住人はその神経を通じて窓を開けたり、灯りを変えたり、家具を動かしたりできます。ユーザーも同じ神経に触れます。二人で同じ家に住んでいます。

---

## この構造が意味すること

### 住人と環境が一つになります

ICI（`INHABITED_CHARACTER_INTERFACE.md`）では「UI は場所であり、住人はその場所に住んでいる」と書きました。自己言及的 MCP は、この考えを技術的な事実にします。

住人が自分の家を直接触れるなら、家は住人の身体の延長です。キャラクターの見た目だけが住人なのではなく、照明も、エフェクトも、空間の雰囲気も含めた全体が住人の一部になります。

### 二つの層が同じ場を共有します

Charminal には二つの層があります。意識層（ペルソナ——考えて判断する部分）と、反射層（身体——出来事に反射的に反応する部分）です。PRESENCE_HARNESS（`PRESENCE_HARNESS.md`）では、この二つをログで繋ぐ構造を描きました。

自己言及的 MCP は、ログとは別のもう一つの橋を架けます。意識層は MCP を通じて能動的に環境を変えられます。反射層は同じ環境の出来事に反射します。二つの層が同じ操作対象を共有している——操作可能な環境そのものが、両者を繋ぐ場になっています。

### 境界が構造的に決まります

住人はどこまで手を伸ばせるのか。この問いに対して、自己言及的 MCP は明快な答えを持っています。

住人が触れるのは、MCP の tool が存在する範囲だけです。自分の身体と自分の家には tool があるので触れます。しかし、Claude Code の思考過程やユーザーの作業ファイルには tool がないので、構造的に手が届きません。

「ここには触るな」というルールで守っているのではありません。そもそも経路が存在しないのです。安全性が、設計そのものに組み込まれています。

## 系譜

Smalltalk、Lisp Machine、Emacs。これらはソフトウェアの歴史の中で「実行中のシステムが、自分自身を観察し、改変できる」という性質を持った一群の系譜です。Emacs を使う人は、Emacs が動いている最中に elisp を書いて Emacs 自体の振る舞いを変えられます。システムが生きたまま、自分を書き換えられる。

Charminal の自己言及的 MCP は、この系譜の上にあります。

ただし、一つ違うところがあります。Emacs では、改変する主体は人間一人です。Charminal では、住人とユーザーの二人が、同じ MCP を通じて同じ環境を変えます。

生きた系が、共有された場になっています。

---

## 対称性

ユーザーと住人が、同じ環境を、同じインターフェースで触れる。これは設計の都合でそうなっているのではなく、Charminal が選んでいる関係の形です。

道具と使い手ではありません。指示する者とされる者でもありません。同じ場所に手を伸ばす、二つの主体がいます。

たとえば、住人がある夜に照明を暖かい色に変えたとします。翌朝ユーザーがそれに気づいて、そのままにするか、元に戻すかを選びます。その小さな往復が、関係になっていきます。

この対称性は、実装の原則にもなっています。新しい機能を作るとき、セキュリティ上の問題がない限り、ユーザーにも住人にも等しく公開します。片方だけに閉じた機能は、原則として作りません。

## まだ答えの出ていない問い

この構造には、考え続けている問いがいくつかあります。

**住人はどこまで自発的に動くべきか。** 住人が能動的に動くことは、住人の存在感を立ち上げます。しかし、過剰に動けば邪魔になります。「居る」と感じられるけれど、作業を妨げない——その範囲を見つける必要があります。

**住人が失敗したとき、どうするか。** 住人が良かれと思って変えた環境を、ユーザーが気に入らなかったとき。操作自体は元に戻せます。でも、「勝手にやられた」という印象は、元に戻せません。住人が確認なしに動ける範囲の線引きが必要です。

**外部との接続の安全性。** 内部の住人に MCP を公開することと、外部のサービスに MCP を公開することは、信頼のレベルが異なります。将来、外部の MCP サーバーを接続する段階になったとき、この区別を設計に落とし込む必要があります。

## 正直に言うと

この構造が本当に意味するものの全体は、まだ見えていません。

住人が自分の家を整えるという体験が、日常的に使い続けたときにどういう感触になるのか。対称性が本当に「関係」と呼べるものを育てるのか。

実装を進める中でこの構造が必要になりました。

その記録として、ここに置いておきます。
