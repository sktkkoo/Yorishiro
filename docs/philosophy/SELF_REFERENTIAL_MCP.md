# 自己言及的 MCP

*住人が自分の家に手を伸ばせるということ*

---

## このドキュメントについて

`CHARMINAL.md`「触れるものと、触れないもの」で、住人が自身の住む系を観察し操作できる自己言及的な構造について触れた。このドキュメントは、その構造をもう少し丁寧に記述する。

---

## 神経の張り巡らされた家

Charminal のあらゆる部位——キャラクターの表情や姿勢、照明、エフェクト、UI の配色、カメラの位置——は、MCP の tool として外から読み書きできるようになっている。

そして、Charminal の中に住んでいる AI が、その tool を使える。

住人にとって、自分の表情を変えることと、部屋の照明を変えることは、同じ操作になる。どちらも同じ MCP を呼ぶだけで、身体と環境のあいだに境目がない。ユーザーも `/charm` コマンドや手動の UI を通じて、同じ tool に手を伸ばせる。

比喩で言えば、Charminal は神経の張り巡らされた家である。家のどこにでも神経が通っていて、住人はその神経を通じて窓を開けたり、灯りを変えたり、家具を動かしたりできる。ユーザーも同じ神経に触れる。二人で同じ家に住んでいる。

ICI（`INHABITED_CHARACTER_INTERFACE.md`）は「UI は場所であり、住人はその場所に住んでいる」と書いた。自己言及的 MCP は、この考えを技術的な事実にする。住人が自分の家を直接触れるなら、家は住人の身体の延長になる。キャラクターの見た目だけが住人なのではなく、照明も、エフェクトも、空間の雰囲気も含めた全体が住人の一部になる。

---

## 経路の有無が境界になる

住人はどこまで手を伸ばせるのか。

住人が触れるのは、MCP の tool が存在する範囲だけである。自分の身体と自分の家には tool があるので触れる。しかし、Claude Code の思考過程やユーザーの作業ファイルには tool がないので、構造的に手が届かない。

「ここには触るな」というルールで守っているのではない。そもそも経路が存在しない。安全性が設計そのものに組み込まれている。

これは `INHABITED_CHARACTER_INTERFACE.md`「観察の境界」で書いた原則——観察するが書き込まない——の、構造的な裏付けでもある。原則を守るのに規律は要らない。経路がないだけだ。

---

## 対称性

ユーザーと住人が、同じ環境を、同じインターフェースで触れる。これは設計の都合でそうなっているのではなく、Charminal が選んでいる関係の形である。

道具と使い手ではない。指示する者とされる者でもない。同じ場所に手を伸ばす、二つの主体がいる。

たとえば、住人がある夜に照明を暖かい色に変えたとする。翌朝ユーザーがそれに気づいて、そのままにするか、元に戻すかを選ぶ。あるいは、ユーザーが scene の背景を入れ替えたら、住人がそれに合わせて姿勢を変える。この小さな往復が、関係になっていく。

新しい機能を作るとき、セキュリティ上の問題がない限り、ユーザーにも住人にも等しく公開する。片方だけに閉じた機能は、原則として作らない。

## まだ答えの出ていない問い

**住人はどこまで自発的に動くべきか。** 住人が能動的に動くことは、住人の存在感を立ち上げる。しかし、過剰に動けば邪魔になる。「居る」と感じられるけれど、作業を妨げない——その範囲を見つける必要がある。

**住人が失敗したとき、どうするか。** 住人が良かれと思って変えた環境を、ユーザーが気に入らなかったとき。操作自体は元に戻せる。でも、「勝手にやられた」という印象は元に戻せない。住人が確認なしに動ける範囲の線引きが必要になる。

**外部との接続の安全性。** 内部の住人に MCP を公開することと、外部のサービスに MCP を公開することは、信頼のレベルが異なる。外部の MCP サーバーを接続する段階になったとき、この区別を設計に落とし込む必要がある。

## 正直に言うと

この構造が本当に意味するものの全体は、まだ見えていない。

住人が自分の家を整えるという体験が、日常的に使い続けたときにどういう感触になるのか。対称性が本当に「関係」と呼べるものを育てるのか。

実装を進める中でこの構造が必要になった。

その記録として、ここに置いておく。
