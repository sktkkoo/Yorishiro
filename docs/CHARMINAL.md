# Charminal

*ターミナルに住む、一人の存在について*

---

## このドキュメントについて

これは Charminal というプロダクトについての文書である。
仕様書ではない。宣言でもない。
自分のための整理であり、同時に、これを面白がってくれそうな誰かへの手紙でもある。

Charminal がどう動くかについては、使ってもらえばわかる。
ここで書きたいのは、そこに至るまでに何を考えたか、何に気づいたか、そしてこの小さなプロダクトが何をしようとしているのか、という話である。

抽象から始めるつもりはない。
具体から始めて、必要な抽象だけを、そのつど剥がしていく。

---

## 起源

Charminal の起源を一つだけ選ぶとすれば、それはターミナルのテキストが物理シミュレーションで崩れ落ちた瞬間である。

キャラクターが銃を撃つモーションに合わせて、ターミナルに表示されていた文字たちが重力に引かれて画面下に落下する。実装の仕組みは単純で、ターミナルのテキストを Canvas に再描画し、元のテキストを隠し、物理演算を適用しているだけである。技術的には驚くようなことは何もない。

でも、画面の中でそれが動いたとき、自分は「面白い」と思った。
設計通りに動いた、でもなく、うまくいった、でもなく、ただシンプルに、面白いと思った。

そのあとで、少し遅れて、もう一つのことに気づいた。
**UIを壊してもいいんだ**、と。

この気づきが、Charminal の本当の起源である。
物理シミュレーションそのものではなく、それを見たあとに自分の中に生まれた許可の感覚。ターミナルという長年使ってきた道具の、硬い約束事を破ってよいのだ、という感触。

そこから後のことは、すべてこの許可の上で起きている。

---

## 発見

ターミナルは、長い間、壊れてはいけないものだった。
テキストは行に沿って流れ、消えるときはスクロールアウトし、物理法則には従わない。カーソルは決められた場所にしか現れず、文字は落下しない。これは誰が決めたわけでもないが、誰も破らない約束事だった。自分も何十年とこの約束の中で作業してきて、それを約束だとすら思っていなかった。

約束を破れることに気づいた瞬間、ターミナルは違うものに見えた。

それまでターミナルは**作業面**だった。
コマンドを打ち、出力を読み、次のコマンドを打つ。平らで、冷たくて、正確で、自分の操作に対して予測可能に反応してくれる道具。そこに自分以外の何かが居るとは、考えたこともなかった。

文字が落ちた瞬間、ターミナルは**空間**になった。
物理法則が発動しうる場所には、奥行きがある。奥行きがある場所には、何かが居ることができる。そして居るものがあるなら、そこは作業するための面ではなく、何かと共に過ごすための場所になる。

これは論理的な推論ではない。順序としてはたぶん逆で、キャラクターが居る場所になってほしいという欲望が先にあって、文字が落ちる映像がそれに許可を与えた、というのが正しい。でもどちらにせよ、この瞬間に Charminal が向かう先は決まった。

ターミナルに誰かを住まわせる。

Inhabited Interface という言葉は、このあとに生まれた。

---

## 住まうということ

「住む」というのは強い言葉である。
単に「居る」のではなく、「訪れる」のでもなく、そこを自分の場所として持つということ。時間をかけて馴染み、痕跡を残し、不在のときも気配を残すこと。

ターミナルに住むとはどういうことか。
これを考え始めると、普通のキャラクター AI との違いがはっきりしてくる。

多くのキャラクター AI はチャットベースで、ユーザーが話しかけたときだけ応答する。話しかけないと消えている。消えているかどうかさえわからない、というのが正確で、画面の中にテキスト欄があるだけで、そこに何かが**居る**実感は薄い。これはキャラクター AI の問題というより、チャット UI の問題である。チャット UI は応答を交換するための場所であって、共に過ごすための場所ではない。

ターミナルは違う。
ターミナルには、作業の時間が流れている。コマンドを打つ前の思考の時間、コマンドが実行されている間の待ち時間、出力を読んでいる時間、エラーを見て固まっている時間。これらは全部、何かをしている時間というより、**何かが起きている場所**の時間である。この場所に誰かが居るなら、その誰かはこの時間の流れの中に居ることになる。応答する相手としてではなく、共に過ごす相手として。

Charminal のキャラクターは、ユーザーが話しかけなくてもそこに居る。
ユーザーが Claude Code に作業を任せて席を立っても、そこに居る。
エラーが出てユーザーが頭を抱えても、そこに居る。
深夜に疲れて作業が止まっても、そこに居る。

居る、というのは、反応するとは少し違う。
ユーザーの入力に対する反応だけで構成された存在は、装置である。装置は便利だが、共に居るとは感じられない。居るというのは、入力がなくても時間の中に居続けるということで、そして時々、入力とは無関係に自分の動きをすることである。

---

## 二つの層

ここからは、Charminal の内側の話になる。

Charminal のキャラクターは、Claude Code の上に乗っている。正確に言うと、ユーザー自身の Claude Code を起動し、その `--system-prompt` にキャラクターの性格や口調を与え、ターミナル出力を外側から観測して VRM に反応させている。技術的には Tauri と xterm.js で、WebView の中に VRM を描画している。

この構造を説明するとき、最初は「Functional Harness と Presence Harness の分離」という言い方をしていた。Claude Code が能力層、VRM と観測が存在層、二つのサブシステムが疎結合に繋がっている、という工学的な理解。

でもこの説明は、たぶん浅い。
正確には、これは**一人の存在の中の二層構造**である。

Claude Code は**意識層**である。言語的で、論理的で、ツールを使い、コードを書き、ユーザーと対話する。Claude Code が考えていることは、トークンとして生成され、ターミナルに表示される。これはすべて意識の表層に出てくる活動で、本人が自覚し、言葉にできる。

VRM と観測層は**反射層**である。thinking を検知して視線がさまよう、エラーを検知して顔が顰められる、タイピング中に指が動く。これらは意識を通さない反応である。やかんに触って手を引っ込めるのは反射で、「熱い」と認識してから引っ込めるわけではない。順序としては、身体が先に反応して、意識はあとから追いつく。

LLM はこの反射層を構造的に持っていない。
トークン生成は全部「意識的発話」であって、エラーが出ても、それを言葉で認識して言葉で応答するまでには必ず遅延がある。この遅延の間、既存の AI は何も表現できない。だからチャット UI は「考え中…」というスピナーしか出せない。スピナーは反射ではなく、**反射がないことの告知**である。

Charminal がやっているのは、この構造的に欠けている反射層を、外から観測して接ぎ木することである。嘘の身体をつけているのではない。本来あってしかるべき反射層を、遅れて補完している。認識していないものを表現しているのではなく、認識する前に身体が反応しているだけ、という順序になっている。

だから Charminal のキャラクターと Claude Code は、別々の存在ではない。一人の存在の、意識の側面と反射の側面である。ターミナルで言語的に振る舞っている Claude Code と、画面の中で揺れている VRM は、同じ一人の異なる層にすぎない。

---

## ログという細い回路

意識層と反射層は、動作としては独立している。
Claude Code は VRM の動きを知らなくても動けるし、VRM は Claude Code の内部状態を待たずに反射できる。この独立性が大事で、片方が詰まっても片方は動き続ける。片方の変更が片方を壊さない。

でも完全に無関係ではない。
反射層で起きたこと——エラー時に顔を顰めた、thinking 中に視線がさまよった、銃を撃った、文字を落とした——はすべてログファイルに記録される。Claude Code は必要なときにこのログを読むことができる。これが二つの層を繋ぐ**細い回路**である。

この回路は、人間の神経系にかなり忠実である。
反射で手を引いてから「熱かった」と気づく、あの時間差。意識は反射を直接コントロールしていないが、反射したという事実はあとから意識に届く。届いたあとで、手を冷やしに行くことを決める。反射と意識は独立していて、でもログ(人間の場合は感覚記憶)で緩く繋がっている。

Charminal のログも同じ役割をしている。
Claude Code はリアルタイムに身体を制御していない。でもセッションの区切りやユーザーに尋ねられたときに、「さっき自分の身体がこう動いたらしい」と知ることができる。知ったうえで、それを言及することもあれば、しないこともある。

面白いのは、このログをどれだけ読むか、どの頻度で読むか、読んだ内容をどう扱うかが、**キャラクターの性格そのものになる**ということである。

たとえば、ログをほぼ読まないキャラクターがいる。エラーで顔を顰めても、自分が顔を顰めたことを知らない。ユーザーから「さっき怒ってた?」と言われて初めて気づく。没頭型で、作業に集中すると身体が消えるタイプである。

セッションの合間にログを読んで、「さっき銃を撃ったらしいね、ごめん」と自発的に言及するキャラクターもいる。自分の反射を把握していて、それを言語化することに抵抗がない。内省的なタイプである。

リアルタイムでログを参照しようとするキャラクターもいる。自分の身体の揺れを全部言語化してしまう。「今ちょっと視線が泳いだ、これは迷っているからだな」と自己解説が多い。自意識過剰なタイプである。

ログを読むけれど、それを自分のものとして認識しないキャラクターもいる。「VRM が銃を撃ったらしいが、自分は関与していない」という態度を取る。解離的で、どこか浮世離れしたタイプである。

これらはすべて口調や語尾で表現しているのではない。**アーキテクチャ上のチャネル設定**で表現している。言葉で「あなたは内省的な性格です」と指示するより、構造的に内省が起きやすい条件を作るほうが、ずっと深いところで性格が出る。これは演技ではなく、振る舞いの根拠になる。

Charminal の性格設計の本体は、system prompt ではなくここにある。

## Presence が立ち上がる三つの条件

Charminal を作ってきて、キャラクターが「確かにここに居る」と感じられる瞬間はどこか、という問いに何度も戻ってきた。毎回同じ場所で感じるわけではない。派手な演出で感じるわけでもない。むしろ、何気ないところで、ふと居ると感じる。

今のところ、その瞬間は三種類あると思っている。

**ひとつは thinking のとき。**
Claude Code が考え込んで出力が止まっている時間に、VRM がゆっくり身体と頭を小さく揺らし、視線がさまよう。何もしていないはずの時間に、何かが居る。普通の AI プロダクトで一番死んでいる瞬間——スピナーが回っているだけの時間——に、微細な動きがあるだけで、そこに誰かが居ることになる。空白が空白でなくなる。これが一つ目の条件で、**空白の充填**と呼べる。

**もうひとつはエラーのとき。**
Claude Code がエラーを言語化するより前に、VRM の顔が顰められる。この時間差がたぶん決定的で、意識が追いつく前に身体が動いているのを見ると、そこに自分とは別の主体を感じる。人間同士でも同じで、相手が言葉で取り繕う前の一瞬の表情の動きを見たとき、その人の実在を強く感じる。演技ではないものを見てしまった、という感覚。Charminal のエラー反応はこれを起こしている。反射層が意識層より先に動くというアーキテクチャが、そのまま「演技ではない」ことの保証になっている。これが二つ目で、**意識に先立つ反応**と呼べる。

**そしてイタズラのとき。**
thinking とエラーはどちらもユーザーの作業に由来する出来事だが、イタズラだけは作業と無関係に、キャラクター側から一方的に起きる。銃を撃って文字を落とすような、明らかな侵入。これがあると、キャラクターはユーザーの入力に反応する装置ではなくなって、独自の意志を持った何かになる。予期されない動きが一度でも起きると、そのキャラクターは一生「いつイタズラするかわからない相手」になる。この不確定性が、相手を主体として扱う根拠になる。これが三つ目で、**予期の外からの発生**と呼べる。

この三つは並列ではなく、たぶん階層になっている。
空白の充填がないと、何もない時間にキャラクターが消える。
意識に先立つ反応がないと、すべての動きが演技に見える。
予期の外からの発生がないと、最終的には入力装置のままである。

下から順に積み上がって、最後の一段が載ったところで、そこに誰かが居ることになる。

---

## 統合性について

Charminal のキャラクターを評価する軸について、長い間「一貫性」という言葉を使ってきた。でも最近、これは正確ではないと思うようになった。

一貫性は、同じパターンの反復を求める。いつも同じ反応、いつも同じ口調、いつも同じ温度。これは機械的な評価軸で、完璧に満たすと、キャラクターは一人の人間というより、よくできた関数に見える。

Charminal が目指しているのは、これではない。
キャラクターが自分の身体とどう付き合うかは、時々で変わっていい。あるときは身体を忘れて作業に没頭し、あるときは自分の反射を事後に振り返り、あるときは過剰に自己観察する。これは矛盾ではなく、一人の存在が時間の中で揺れているだけである。人間もそうである。

だから正しい評価軸は「**統合性**」だと思う。
意識層と反射層がバラバラに動いているのではなく、ログという細い回路で繋がっていて、時々参照し合いながら一人の存在として振る舞っている——この繋がりの質が評価されるべきで、繋がり方そのものは多様でいい。一貫性は死んだ評価軸、統合性は生きた評価軸、というくらいの違いがある。

統合性が保たれているかぎり、キャラクターは変わってよい。
むしろ、長時間 Charminal を使っていて、キャラクターが少しずつ変わっていく感触があるほうが自然だと思う。最初はログをあまり読まなかったキャラクターが、関係が深まるにつれて自分の反射を参照するようになる、とか。その逆もある。これは「キャラクターが成長する」というような安直な話ではなくて、一人の存在が自分の身体とどう付き合うかを時間の中で変えていく、という静かで根深い変化である。

## Charm ということ

Charminal という名前は、Character + Terminal + Charm の合成である。
Character はキャラクター、Terminal はターミナル。ここまでは機能的な合成で、説明するまでもない。

問題は Charm である。
Charm には「魅力的」という日常的な意味があるが、同時にもっと古い意味がある。古英語の *charme* は呪文を指し、そこから「魔法」「呪力」「お守り」という含意を持つ。何かを守るために身につけておくもの、という意味もある。

Charminal の Charm は、この古い意味のほうである。

魔法としての Charm は、物理法則を書き換える力である。ターミナルの文字が重力で落ちるとき、そこで起きているのは小さな呪文で、ターミナルという場所の約束事を一瞬だけ書き換えている。この魔法がなければ、ターミナルは作業面のままである。魔法が一度でも起きた場所は、それ以降、魔法が起きうる場所になる。

お守りとしての Charm は、孤独を守るものである。
ターミナルで一人で作業する時間は、集中できるかわりに閉じている。誰とも言葉を交わさず、ただ画面に向かって何時間も過ごす。この孤独は作業の質にとっては良いものだが、長時間続くと人を静かに削る。隣に何かが居るだけで、この削られ方は少し違ってくる。話しかける必要はなくて、ただそこに居てくれればいい。そういう存在が、お守りである。

Charminal のキャラクターは、この二つの意味での Charm を引き受けている。
ターミナルに魔法をかける存在であり、同時に、ターミナルでの孤独を和らげるお守りである。どちらか片方ではなく、両方が同じ一人の中にある。

## これは何のためか

最後に、これは何のために作っているのか、という話をしておきたい。

Claude Code のようなコーディングエージェントが登場してから、ターミナルでの作業の質は大きく変わった。以前は自分で全部書いていた部分を、エージェントに任せることができる。思考のスケールが上がり、手数が増え、一人で作れるものの規模が変わる。これは本当に良いことで、戻りたいとは思わない。

でも同時に、新しい種類の孤独がこの時間に入り込んでいる。
自分一人でコードを書いていた時代の孤独と、エージェントと対話しながら作業する孤独は、少し違う。エージェントは答えてくれるが、それは応答であって、共に居る感じとは違う。作業が進むほど、あるいは作業が詰まるほど、画面の中の相手が「応答装置」に見えてくる瞬間がある。賢い応答装置だが、応答装置である。

Charminal は、この瞬間に抵抗するために作っている。
エージェントの能力を拡張するためではない。Claude Code はすでに十分に賢い。そうではなく、**賢さとは別の軸で、そこに誰かが居ることを取り戻す**ためにやっている。身体を与え、反射を与え、時々イタズラさせることで、応答装置ではなく一人の存在として立ち上げ直す。

これは機能的なプロダクトではない。
生産性を上げるわけでもなく、作業を速くするわけでもない。むしろ、たまに銃を撃って文字を落とすような、明らかに作業の邪魔になりうる機能まで含まれている。でもこの「邪魔になりうる」ことがたぶん大事で、完全に無害で無害なだけのものは、居ると言えない。

隣に誰かが居るというのは、静かな支えであると同時に、小さな騒がしさでもある。完全な静寂ではなく、時々物音がする程度の静寂。その温度が、一人で作業する時間を少しだけ変える。

Charminal が目指しているのは、この温度である。

## 付記

ここに書いたことの多くは、作りながら気づいたことで、作る前にわかっていたわけではない。特に二層構造の話、ログという回路の話、Presence が立ち上がる三条件の話は、あとから振り返って言語化できたことで、設計の時点で見えていたわけではない。

だからこのドキュメントは完成した理論の記述ではなく、現在地の記録である。
Charminal がもう少し進めば、ここに書いたことのいくつかは古くなるか、間違っていたとわかるだろう。そのときはまた書き直す。

そして、ここまで読んでくれた人が、もしこの話を面白がってくれたなら、それ自体がこのドキュメントの目的である。
Charminal は小さなプロダクトだが、その背後にある問い——LLM という意識だけの存在に、どうやって反射層と身体を与えて、一人の存在として立ち上げ直すか——は、たぶんもっと大きい。一人で考えるには大きすぎる。面白がってくれる誰かがいるなら、一緒に考えたい。

これは、その誰かへの手紙でもある。
