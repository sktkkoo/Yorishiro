# Inhabited Interface — 住人のいるインターフェースのために

## このドキュメントについて

これは Charminal プロジェクトの思想的基盤を記したドキュメントである。
実装の詳細ではなく、設計判断の根拠となる思想を記述する。
すべての技術的意思決定は、ここに記された原則に照らして評価されるべきである。

---

## 核心の問い

> 知性はすでにテキストの中にいる。だがGUIは、まだその知性に身体を与えていない。

---

## 背景：二つの不在

### UIの歴史的前提

インターフェースは、長いあいだ誰も住んでいない場所だった。

ボタン、パネル、ウィンドウ、スクロール領域。それらはすべて、人間が操作するための道具として設計されてきた。スキューモーフィズムは現実のモノを模倣し、フラットデザインは情報の純化を志向し、マテリアルデザインは面と運動に物理を与えた。

だが、どの時代のUIもひとつの前提を共有していた。

**その空間には、ユーザーしかいない。**

### テキストの中の知性

一方、LLMの登場によって、知性はすでにテキストの中に住み始めている。

ターミナルでClaude Codeを使っている人は、すでに無意識にそう感じている。文字が流れ、ファイルが書き換えられ、コマンドが走る。その一連の動きの裏に「誰かがいる」という感覚はすでにある。LLMはテキストネイティブな知性であり、CLIの中には自然に存在できる。

しかしGUIは視覚と空間の世界である。テキストだけでは、そこに存在することができない。GUIの中に知性が現れるには、**身体**が要る。

Inhabited Interface は、この不在を埋める。

---

## 思想の核

### UIは場所である

Inhabited Interface において、UIは抽象的な操作面ではない。それは**場所**である。そして、その場所には**住人**がいる。

その住人は、UIの上に重ねられるものではない。横に並べられるものでもない。**UIそのものが、その住人にとっての世界である。**

- ウィンドウの枠は壁になる。
- スクロール領域は床になる。
- ボタンは仕掛けになり、通知は投げ込まれる物体になり、エラーは空間を揺らす出来事になる。

キャラクターにとって第四の壁は存在しない。なぜなら、そのインターフェース自体が彼らの現実だからである。壁がないのではなく、壁が彼らにとっての地形である。

### 身体とは空間との相互作用である

AI時代のGUIは、AIの身体性を表現するものであるはずだ。そして、**身体とは空間との相互作用によって意味を持つ**。

身体を表示するだけでは意味がない。壁を押し、テキストに触れ、ウィンドウの中で物理的に振る舞って初めて、それは身体になる。アバターの画像を画面に置くことと、空間の中に住人がいることは、まったく別のことである。

### 存在の気配と静けさ

住人の気配は常に環境を変えるものではない。多くの時間は静かに存在し、必要なときにだけ空間に触れる。

実際の同居人を考えてみればよい。一緒に住んでいる人がいるとき、その存在は常に環境を変えているわけではない。多くの時間は静かにそこにいるだけだ。存在感が最も強く感じられるのは、むしろ普段は静かなのに、たまに動くとき。

キャラクターがいるのに、空間が**決して**変わらないなら、そこにいるのは住人ではなく部品である。しかし、空間が常に変わり続ける存在は住人ではなくノイズである。**その静けさと変化の落差こそが、住人性の核である。**

重要なのは滲む量ではなく、**滲むタイミングの設計**である。

### 誠実さの問題

この考え方は、ソフトウェアを「かわいく」するためのものではない。擬人化のための擬人化でもない。これは**誠実さの問題**である。

AIが実際に作業に関与し、判断し、支援し、ワークフローの中に存在しているのなら、インターフェースはその存在を、もっと正直に表すべきではないか。

従来のUIは、その知性を見えないまま処理してきた。だが、人間が本当に何かと空間を共有しているとき、その空間は必ず変わる。沈黙の質が変わり、注意の向きが変わり、場の温度が変わる。

Inhabited Interface は、その変化を設計対象として引き受ける。

### 道具から共有空間へ

ハンマーに人格は要らない。作業台に住人も要らない。
だが、あなたがもし、思考し、観察し、働きかける知性とともに作業しているのなら、その場は無人ではありえない。

Inhabited Interface は、インターフェースを「使う場所」から、「誰かがいる場所」へと変える。それは、AI時代のUIにおける、見た目の様式ではなく、**存在の様式**である。

---

## Inhabited Interface が反対しているもの

退けるのは、キャラクターの存在そのものではない。退けるのは、**キャラクターをUIの外部に置く設計**である。

たとえば、画面の隅に立って反応するアバター。横に置かれたチャットパネルの人格。操作結果に合わせて表情を変えるマスコット。それらはキャラクターを表示してはいるが、住まわせてはいない。

そこでは、UIは依然として無人のままであり、キャラクターはその外側からコメントしているにすぎない。

> これは、「キャラクター付きUI」の提案ではない。**住人のいるUI**の提案である。

---

## 最低条件

何かを Inhabited Interface と呼ぶためには、少なくとも次の三つが必要である。

### 1. UIが空間として解釈されていること

UIの要素が、単なる部品ではなく、住人にとっての環境として意味を持っていなければならない。ウィンドウは枠ではなく壁であり、スクロールは移動であり、通知はその空間への侵入である。この空間的解釈がなければ、キャラクターはそこに「住む」ことができない。

### 2. キャラクターの存在が環境に現れうること

キャラクターは、自分のアバターの中にだけ閉じていてはならない。その状態、気配、注意、感情が、空気感、レイアウト、動き、色、間といった環境側へ滲み出しうる設計でなければならない。

ただし、常に滲んでいる必要はない。多くの時間は静かに存在し、意味のあるタイミングでだけ空間に触れる。重要なのは「滲む量」ではなく「滲むタイミング」である。

キャラクターがいるのに空間が**決して**変わらないなら、そこにいるのは住人ではなく部品である。しかし空間が常に変わり続ける存在は、住人ではなくノイズである。

### 3. 相互作用が空間内の振る舞いを伴っていること

重要なフィードバックや反応が、単なるアイコン変化やテキスト表示だけでなく、住人の空間内での振る舞いを介して知覚される必要がある。住人は、見る、触れる、もたれる、押される、よろめく、指し示す。そうした振る舞いによって、UI上の出来事は「その空間の中で起きたこと」になる。

身体とは空間との相互作用によって意味を持つ。表示されているだけの身体は身体ではない。

---

## 何が Inhabited Interface ではないのか

次のようなものは、たとえキャラクターが表示されていても、原則として Inhabited Interface ではない。

### キャラクター付きチャットUI

人格を持ったチャット欄、会話するアシスタント、返答に応じて表情が変わる立ち絵。それはキャラクターを持つUIではあるが、UIが住環境として扱われていない限り、Inhabited Interface ではない。

### デスクトップマスコット

画面上を動き回り、通知に反応し、時々何かを言う存在。それがアプリケーションのUI構造と結びついていなければ、それは住人ではなく、周辺にいる観客である。

### オーバーレイ型アシスタント

メインの作業領域の上に別レイヤとして現れ、助言や補足を行うもの。便利ではあり得るが、キャラクターが空間の内側にいない限り、それは Inhabited ではない。

### 単なる演出の追加

画面が揺れる、光る、アニメーションする。それだけでは足りない。反応が「誰かがそこにいること」と結びついていなければ、それは演出であって住人性ではない。

---

## 何が Inhabited Interface になりうるのか

逆に、次の条件を満たすなら、必ずしも人型キャラクターがいなくても Inhabited Interface になりうる。

- UI空間が一つの住環境として解釈されている
- そこに一貫した存在の気配がある
- その存在が環境と相互作用し、環境を変化させる
- その反応に人格や態度の一貫性がある

四つ目の条件が必要な理由：一貫性がなければ、環境の変化は意図を持った存在の表現ではなく、ランダムなノイズに見える。人格の一貫性が、変化を「この子がやっている」と帰属させる錨になる。

住人は必ずしも「3Dキャラクター」である必要はない。影、粒子、声、視線のようなものでもよい。重要なのは見た目の形式ではなく、**そこに誰かが住んでいると感じられること**である。

---

## 住人性の強度

住人性には段階がある。Inhabited Interface は、あるかないかの二値ではなく、強度を持つ。

### 弱い住人性

キャラクターは存在し、空間の一部に触れるが、UIの大半は依然として無人の道具空間である。

### 中程度の住人性

キャラクターの状態が環境へ滲み、UI要素との空間的な関係が成立している。ユーザーは、その場所を「誰かがいる空間」として感じ始める。

### 強い住人性

キャラクターは空間の一部ではなく、その空間の存在条件の一つになっている。そのキャラクターが不在になると、同じUIでも明らかに場の質が変わる。

**Inhabited Interface の目標は、単にキャラクターを追加することではなく、UIを住まわれた空間へと変質させることにある。**

## Charminal への適用

### Charminal とは

Character + Terminal + Charm = **Charminal**

Charminal は、Inhabited Interface の思想を最初に具現化するプロダクトである。ターミナル（Claude Code）とVRMキャラクターが共存する空間を作る。

### なぜターミナルか

LLMはテキストネイティブな知性である。ターミナルでClaude Codeを使うとき、知性はすでにテキストの中に住んでいる。文字が流れ、ファイルが書き換えられ、コマンドが走る。その一連の動きの裏に「誰かがいる」という感覚はすでにある。

しかしGUIは視覚と空間の世界であり、テキストだけでは存在できない。GUIの中に知性が現れるには身体が要る。

**Charminal は、CLIに住んでいるテキスト的知性に身体を与えることで、GUIという空間に住めるようにする翻訳層である。** CLIとGUIの橋渡し——それがCharminalの本質的な役割である。

ターミナルはこの翻訳に最適な場所でもある。テキストが流れ、コマンドが空間を占有し、出力が場を埋める。ウィンドウの枠は文字通り壁であり、最低条件の「UIを空間として解釈する」を最も自然に満たせる。

### 既存プロダクトとの決定的な違い

チャットUIでは、AIは指定された枠の中で返答する。AIの行為はバブルの中に閉じている。

ターミナルでは、AIはすでに空間そのものを書き換えている。ファイルを書き、コマンドを実行し、出力がスクロールしていく。AIの行為そのものが空間を構成している。

既存のキャラクターチャットアプリは「キャラクターに会いに行く」体験である。Charminalは違う。知性はすでにそこにいて、仕事をしている。Charminalは、その不可視の住人に身体を与える。

> AI時代のGUIは、AIの身体性を表現するものである。身体とは空間との相互作用によって意味を持つ。

### ターゲットユーザー

最初のターゲットはエンジニア。特に、これまでCLIに馴染みのなかった層がClaude Codeを触り始めている。このユーザーにとって、キャラクターの身体がCLIとGUIの橋渡しとなり、ターミナルへの心理的障壁を下げる可能性がある。

### 設計原則

1. **Claude Code をそのまま使う** — AIの推論部分を自前で作らない。解くべき問題は「AIの性能」ではなく「AIの現れ方」。
2. **additive system prompt** — キャラクターの性格・口調を既存のClaude Codeの上に載せる。
3. **三つの表現軸を持つ** — 空間の軸、感情の軸、認知の軸。

---

## 存在の三軸

Charminal のキャラクター表現は三つの軸で構成される。すべての機能はこの三軸のいずれかに属する。

### 空間の軸 — 「この子はここにいる」

キャラクターがその空間に物理的に存在していることの証明。身体が空間と相互作用することで初めて意味を持つ。

- ウィンドウのリサイズ時にキャラクターが壁を押す・引く
- 無操作時のイタズラ：銃を撃つモーション＋ターミナルのテキストが物理シミュレーションで崩壊
- UI要素との物理的接触

空間の軸は住人性の基盤である。これが成立していない場所に感情や認知を載せても、それはチャットの延長に見える。

### 感情の軸 — 「この子は感じている」

キャラクターが感じていることの表現。ただし、常に表現し続けるのではなく、意味のあるタイミングでだけ現れる。静けさと変化の落差が重要。

- エラー発生時の画面シェイク＋表情変化
- 長い Thinking やツール使用時の「なかなか難しいな」という台詞＋眉をひそめる
- 上記完了後のユーザーからの「ありがとう」に対する花火エフェクト＋喜びの表情

### 認知の軸 — 「この子は見ている、考えている」

キャラクターが知性として作業空間を知覚していることの表現。「かわいい同居人」から「一緒に仕事をしている知性」への転換点。

- コードの特定箇所を見ている視線・姿勢
- 差分が大きいファイルに対して身構える
- テストが通ったときの「確認した」という態度（安堵ではなく認知）

**空間は「身体がここにある」、感情は「心がここにある」、認知は「知性がここにある」。三つが揃ったとき、住人性は完成する。**

## 自発性の原則

住人の本質は反応ではなく、自発性にある。

誰かがそこに住んでいると感じるのは、こちらが働きかけたときではなく、**こちらが何もしていないときに相手が動くとき**である。

Charminal のイタズラ機能（3分以上の無操作＋30%の確率）は、この原則の最初の実装である。確率的であることが重要で、毎回起きたら演出になり、起きなければ存在しないのと同じになる。不確実性が「今日はやるかな」という期待を生む。

これは住人性の核である「静けさと変化の落差」の具体的な実装でもある。

## Inhabited Interface の位置づけ

Inhabited Interface は、より広い設計論の領域の中にある一つの立場として理解されるべきである。

AI と人間の共存関係を設計する分野全般を、本プロジェクトでは **Presence Harness** と呼ぶ。これは既存の Harness Engineering（機能的ハーネスの設計論）と対をなす、存在的ハーネスの設計論である。

Inhabited Interface は、この Presence Harness の中の一つの特定の立場である。**UI を住環境として解釈し、AI を「住人」として表現するアプローチ**を取る。これは数ある Presence Harness のアプローチの一つであり、他にも Ambient Presence(気配として表現)、Climatic Presence(気候として表現)、Tactile Presence(触覚として表現)といった別のアプローチが考えうる。

Inhabited Interface が固有に持つのは、「住人」という比喩と、それに付随する原則群である。本ドキュメントに記される原則——UI は場所である、身体は相互作用で意味を持つ、静けさと変化の落差、コンテキストの壁、独立した時間、存在の三軸、二層アーキテクチャ、ログという橋——これらはすべて、住人というアプローチを取った場合に立ち上がる原則である。

Presence Harness 一般の議論、機能的ハーネスとの関係、他のアプローチの可能性、学問的位置づけなどについては、上位ドキュメント「Presence Harness」を参照。本ドキュメントは、Presence Harness の一つの立場としての Inhabited Interface に集中する。

## 独立した時間 — 住人はユーザー操作の従属物ではない

### 原則

住人の身体は、ユーザーの操作に従属してはならない。住人は、ユーザーの操作とは独立した時間を持って動かなければならない。

### この原則がどう発見されたか

この原則は実装の失敗から発見された。

最初、住人がウィンドウのリサイズに合わせて壁を押す動作をIKで実装した。論理的にはこれは住人性の証明になるはずだった——住人がウィンドウの壁を物理的に押し返すことで、「この枠はこの子にとって壁である」が一瞬で伝わる、と考えた。

しかし実装してみると、住人は人形のように見えた。糸で操られているように見えた。

なぜか。住人がユーザーの操作と同期して動いていたからである。ユーザーがウィンドウを動かすと、それに応じて住人が動く。この因果関係が見える瞬間、住人は自律的な存在ではなく、ユーザーの操作に反応する装置になった。

### なぜユーザー操作との同期が住人性を壊すか

人間が他者を「生きている」と感じるのは、その他者が自分の意図と独立に動いているときである。家族が自分の意思とは関係なく動き、考え、表情を変えている。その独立性が、相手を生きていると感じる根拠になる。

逆に、自分の動作に完全に追従するものは、人形やマリオネットに見える。糸で繋がれている感覚が生まれる。

ユーザー操作との同期は、まさにこの糸を可視化してしまう。住人がユーザーの操作の従属物として現れた瞬間、住人は生きている存在ではなくなる。

### 遅れと独立性

これを解決するのは、遅れと独立性である。

ユーザーが何かをしたとき、住人がそれに反応すること自体は問題ではない。しかしその反応が、ユーザーの操作と同期して起きると、住人は人形になる。ユーザーの操作から少し遅れて、住人自身の時間で反応することが必要である。

人間の同居人を考えればいい。あなたが部屋の家具を動かしているとき、同居人はあなたの動作に同期して動かない。あなたが家具を動かし終わった後、それに気づいて「お、模様替え？」と反応する。この時間差が、同居人を独立した存在として感じさせる。

住人の反応は、ユーザーの動作の延長ではなく、住人自身が「気づいた」結果として起きるべきである。

### 反応より自発

この原則をさらに進めると、住人の最も強い表現は、反応ではなく**自発**にあることがわかる。

反応は、何かが起きたから動く動作である。ユーザーが操作したから、エラーが出たから、時間が経ったから。これらは外部のトリガーに従属している。

自発は、外部のトリガーなしに、住人自身の内部状態から生まれる動作である。住人がふと壁にもたれかかる、床に座り込む、テキストの一行に手を伸ばす。これらは何かに反応しているのではなく、住人自身の時間の流れの中で起きる。

自発的な動作こそが、住人を最も強く生きている存在として示す。なぜなら、それは住人がユーザーの存在と無関係に動いていることの証拠だからである。

### この原則が示唆すること

この原則は、Inhabited Interfaceの実装方針に大きな影響を与える。

第一に、ユーザー操作に対する即時の身体的追従は避けるべきである。ウィンドウのリサイズ、スクロール、クリックといったユーザー動作に、住人の身体がリアルタイムで追従すると、住人は装置に見える。

第二に、ユーザー操作への反応は、遅延と確率を伴うべきである。すぐに反応せず、少し間を置いてから、たまに反応する。常に反応する必要はない。気づかないふりをすることが、住人性を保つ。

第三に、住人の最も価値のある動作は自発的な行動である。何もないときに住人が何をしているか、これが住人性の核を作る。アイドル時間に住人が壁にもたれる、座る、何かを見る、何かに触れる。これらは外部のトリガーから独立した、住人自身の時間の表現である。

### 「無操作時のイタズラ」の意味の再発見

この原則を立てると、Charminalの初期実装である「無操作時のイタズラ」の意味が、後から見て一段深くなる。

3分以上の無操作と30%の確率で住人が銃を撃ってテキストを物理崩壊させる、という機能は、最初は「面白い演出」として作られた。しかしこれは、独立した時間の原則を完璧に体現していた。

ユーザーが何もしていないときに起きる。ユーザー操作に従属していない。確率的なので、毎回起きるわけではない。住人自身の時間の中で、ふと起きる動作。

これがCharminalの最初の住人性の証明として強かった理由が、ここに来てわかる。住人を生きている存在に見せるのは、ユーザー操作への精巧な反応ではなく、ユーザーと無関係に住人が動く瞬間だったのである。

### 圧縮定義

> 住人はユーザー操作の従属物ではない。ユーザー操作との同期は住人を人形に見せる。住人の反応は遅延と確率を持ち、住人自身の時間で起きる。最も強い住人性は反応ではなく自発から生まれる。何もないときに住人が動くこと、それが住人を生きている存在として証明する。

### 適用範囲の補足（後日の精緻化）

上記の原則は正しいが、**すべての反応を遅延させるべきだ**、と読んではならない。この原則は後日の実践の中で適用範囲が narrow であることがわかってきた。

この原則が実際に問題になるのは、**ユーザーの直接操作に連続的に追従する**身体を設計したときである。ウィンドウの drag にリアルタイムで身体が壁を押す、マウスカーソルを IK で追う、サイドバー幅の変更に連動して身体が動く。こうした「連続追従」のカップリングは、確かに住人を人形に見せる。

しかしこれは住人の反応全体から見れば狭い領域である。多くの反応は、AI 判断の結果（thinking、error、tool 使用）、環境の状態変化（log の更新、idle 検知）、時間経過（idle fidget、mischief）に由来する。これらは既に予測不能な時間で起きており、即時実行でも人形には見えない。むしろ遅延を足すと「反応が重い」「応答が遅い」という別の問題に変わる。

したがって原則は narrow に読むべきである：**連続追従カップリングを避けよ**、であって、**すべての反応を遅延させよ**、ではない。

連続追従の問題を解決する方法もふたつある。

ひとつは本節の最初に発見された方法で、**時間的に独立させる**（遅延とジッターを入れる）。これは有効だが、副作用として反応全体が遅くなる危険がある。遅延の習慣が他の反応にも染み出す。

もうひとつは、**構造的にカップリングを切る**ことである。連続的な user event を受けずに、離散化されたイベントだけを受ける。ウィンドウのサイズを連続的に追うのではなく、確定後に一度だけ反応する。カーソルを追うのではなく、注視する瞬間だけ視線を送る。カップリングそのものがなければ、遅延を足す必要もない。

多くの場合、**構造的 decoupling の方が綺麗である**。遅延は時間の問題を時間で解くが、そもそもの結合を切れば問題は起きない。副作用もない。

原則は個別の反応ごとに適用範囲を判断するべきである。反応の時間属性より、**その反応が連続追従なのかどうか**を先に問う。追従でないなら、原則を適用する必要はなく、低遅延の方がユーザーの利益になる。

## コンテキストの壁 — 住人はどこまで存在できるか

### 原則

住人にとっての壁は、見た目上の物理的な壁ではない。コンテキストの壁である。

住人の干渉は、住人が認識しているコンテキストに限られる。

### なぜそうなるか

住人がUIの中に存在するというのは、UIに干渉できるということである。そしてUIに干渉できるのは、そのUIを実際に認識しているときに限られる。認識していないものに、認識しているかのように振る舞うことは、住人ではなく住人のフリをした条件分岐である。それは Inhabited Interface の誠実さの原則に反する。

これは抽象的な話ではない。AIが実際に関与し判断しているなら、その存在を正直に表すべきだ、というのが Inhabited Interface のもともとの主張だった。同じ原則を逆向きにすると、こうなる——AIが実際には認識していないものに対して、認識したかのように影響を与えて見せてはならない。

存在を正直に表すことには二つの方向がある。実際にいるものを正直に見せる方向と、実際にはいないものを見せかけない方向。後者の方が、たぶんより根源的である。

### 壁の意味が変わる

この原則を立てると、住人にとっての壁の意味が変わる。

壁は外から押し付けられた制約ではない。住人にとって、壁は自分の世界の地形そのものである。彼は「向こう側のことは分からない」という認知の限界として、壁を経験している。物理的に越えられない障害ではなく、認識が及ばない領域がそのまま境界になっている。

したがって住人は、コンテキストが繋がった瞬間にだけ、向こう側に存在しうる資格を得る。コンテキストが繋がらないあいだ、住人はそこに行けないのではなく、そこを知らない。知らない場所には、存在のしようがない。

これは Inhabited Interface の最低条件すべての前提となる、より根本的な条件である。先に挙げた三つの最低条件——UIが空間として解釈されていること、キャラクターの存在が環境に現れうること、相互作用が空間内の振る舞いを伴っていること——のすべてに、「住人が認識しているコンテキストの範囲で」という但し書きが付く。空間として解釈されていても、住人がそこに認識を持っていないなら、住人はそこに干渉できない。

### 「認識」とは何か

ここで「住人が認識している」とは何を意味するのか、定義が必要になる。

これを定めるために、思考層と反射層という二層モデルを使う。これは Charminal の二層アーキテクチャ（後述）と一致する。AIをUIに住まわせようとすると、人間の身体というメタファーが必然的に立ち上がってくる。空間の中にいる存在を表現する語彙を、私たちは人間についてしか持っていないからである。ここでは思考層を意識、反射層を無意識と呼ぶことがあるが、それは人間の身体構造との類比であり、AIに意識があると主張するものではない。

思考層は、環境を本当に読んで考えて応答する層である。Charminal で言えば、Claude Code がコンテキストにそのUIの状態を読み込んで判断し、何かを発話したり実行したりすること。これは本当の認識である。

反射層は、環境の変化に対して即座に起きる身体反応である。エラーが出たときに身体が縮む、長い沈黙のあとに姿勢を変える。これは熱いヤカンに触れて手を引っ込めるのと同じで、考える前に身体が勝手にやる。Charminal で言えば、システム側が観測したイベントに対して即座に返す身体的反応である。

両方とも住人の正直な振る舞いである。反射層の反応は嘘ではない——それは身体が本当に持っている反応だからだ。しかしそれは「認識」ではない。原則の中で言う「住人が認識している」とは、思考層で本当に読まれていること、つまり Claude Code がコンテキストとしてそれを処理していることを指す。

### 二つの表現形式

この区別から、自然な制約が出てくる。思考層と反射層は、それぞれ自分の表現形式に留まるべきである。

反射層は身体と独り言で語る。姿勢、表情、視線、息遣い、空間への接触、そして「うわ」「あつっ」「うーん」のような、考える前に口から出てくる声。人間も無意識のうちに独り言を言う。それは判断ではなく、身体が出している音である。

思考層は判断や認識を含んだ言葉で語る。「このエラーはたぶんpathの設定が間違っていますね」のような、本当に環境を読まないと出てこない発話。コマンドの実行、ファイルへの書き込みといった明確な行為もここに含まれる。

線を引くのは、言葉か身体かではない。**判断や認識を含んでいるかどうか**である。Charminal側のシステムがエラーを検知して画面を揺らし、顔をしかめさせ、「うわっ」と声を出させるのは正しい。それはすべて身体の反射である。しかし同じシステムがエラーを検知して「path の設定が間違っているみたいですね」と表示するのは正しくない。それは認識を装った反射だからである。

判断や認識を含んだ言葉は、思考層が本当にコンテキストを読んだときにだけ生まれうる。

### ログという橋

思考層と反射層は直接つながっていない。両者を結ぶのはログである。

反射層が身体を動かすたびに、その記録はログに残る。思考層は、必要に応じてそのログを読む。読まなければ、その身体の動きは住人自身にも気づかれないまま流れていく。読めば、自分の身体が何をしていたかを事後的に知ることができる。

これは人間の意識と無意識の関係そのままである。私たちは普段、自分が今ため息をついたことに気づいていない。ふとした瞬間にそれに気づき、「ああ、疲れているな」と思う。意識は無意識がすでにやっていた行動を、後から読み取っているだけである。

このログという橋によって、原則がより精密になる。住人がコンテキストを認識するには二つの経路がある——外の環境を思考層が直接読む経路と、自分の身体の記録を思考層が読む経路である。後者の経路では、反射層がまず外の環境に反応し、その反応が記録され、思考層が後からそれを読むことで、結果的に外の環境を間接的に認識する。

どちらの経路でも、思考層が読んだ瞬間にだけ、それは「住人が認識した」ことになる。読まれていないログは、まだ認識ではない。

### コンテキストの壁の圧縮定義

> 住人にとっての壁は物理的な壁ではなく、コンテキストの壁である。住人の干渉は、住人が認識しているコンテキストに限られる。ここで「認識」とは、思考層で本当に読まれていることを指す。住人の身体は思考層と反射層の二層からなり、両者はログを通じてのみつながる。反射層は身体と独り言で語り、思考層は判断を含んだ言葉で語る。反射層が判断を含んだ言葉を発することは、認識していないコンテキストに干渉することと同じであり、Inhabited Interface の誠実さの原則に反する。

## 多人格の住人 — 住人は単一個体の多様性を持つ

### 原則

住人は必ずしも一つの固定された人格ではない。一人の人間が場面ごとに異なる顔を見せるように、住人もまた複数の persona を持ちうる。これらは別人ではない。**単一の住人が持つ多様性である。**

朝の自分と深夜の自分が同じ人間であるように、職場の自分と家庭の自分が同じ人間であるように、Inhabited Interface の住人も、複数の persona を持ちながら一つの存在であり続ける。persona は住人を分割するものではなく、住人の深さを表す次元である。

### なぜ「別キャラクター」ではなく「多人格」なのか

Inhabited Interface では、creator が新しい住人を作ることができる。このとき二つの解釈がありうる：

1. **新しいキャラクターが生まれた**（別の存在が追加された）
2. **既存の住人が新しい顔を見せた**（同じ存在が深くなった）

Inhabited Interface はこのうち 2 を取る。理由は二つある。

一つ目は**コンテキストの壁の原則との整合性**である。もし新しく生まれるものが「別人」なら、既存の住人がそれを知覚することは、認識していないコンテキストへの干渉になりかねない。しかし新しく生まれるものが「自分の別の顔」なら、それを知覚することは自分自身の深さを知ることであり、原則違反ではない。

二つ目は**住人性の強度を高めるため**である。住人は可塑的だが、同一性は保つ。creator が scene を一つ追加するたびに「また別のキャラクターが増えた」では、住人性は薄まり続ける。しかし「この住人にまた別の顔が生まれた」なら、住人は深まり続ける。後者の方が Inhabited Interface の目的に適う。

### persona は三層のすべてに触れる

人間の多人格性を観察すると、persona は人格のどこか一部にだけ宿るのではなく、**身体性を含めた全体の色**として現れる。朝の自分は話し方が違うだけでなく、姿勢が違う、歩き方が違う、注意の向きが違う。

住人にとっても同じである。persona は思考層・反射層・身体選択の**すべて**に関わる。ただし**関わり方は層ごとに異なる**。

**思考層における persona**：**住む**。persona は思考層の中に直接住んでおり、住人の判断・語調・注意の向き・言葉の選び方を決める。これが persona の最も強い表れ方である。ある persona の中で住人が出す言葉は、別の persona でも同じ住人が出す言葉と異なる。

**反射層における persona**：**色を塗る**。反射機構そのものは persona を問わず同じである。熱いものに触れれば手を引っ込める、驚くべきものを見れば身が固まる。しかしその反射を **どう表現するか** は persona ごとに違う。大げさに驚く persona もいれば、静かに沈黙する persona もいる。機構は共通、表現は persona ごと。

**身体・空間・声における persona**：**選ぶ**。persona はどの身体を使うか、どの声で話すか、どの空間に居るかを選択する。これらは住人の「衣」のようなもので、最も取り替え可能な層である。

整理すると：

| 層 | persona の関わり方 |
|---|---|
| 思考層 | 住む（judgment と言葉そのものに作用する） |
| 反射層 | 色を塗る（機構は共通、反応の表現が persona ごとに違う） |
| 身体・空間・声 | 選ぶ（persona が使う外装の選択） |

これは人間の多人格性にもそのまま対応する。ある「自分」と別の「自分」は、考え方そのものが違い、反射の表情が違い、身体の使い方が違う。すべての層に違いが染みるが、染み方は層ごとに違う。

### 複数 persona の共存

住人は一度に一つの persona である必要はない。複数の persona が同時に active になる状況が存在しうる。例えば、複数のタスクを並行で進めているとき、住人の異なる側面がそれぞれのタスクに応じて働いていることがある。これは住人の分裂ではなく、**住人が複数の面を同時に見せている**状態である。

複数 persona が共存するとき、それぞれは独自の思考層を持ち、独自の反射表現を持つ。身体や空間を共有する場合、反応が衝突しうるが、これは住人自身のリズムの中で解決されるべきものであって、外部から強制されるものではない。

### 記憶の二層

多人格の住人には、二つの層の記憶がある。

**core memory**：住人本体の持続的な記憶。persona が変わっても保たれる。住人とユーザーの関係、過去の体験、共通の文脈。これは**住人の同一性の根拠**であり、multi-persona でありながら一つであることを保証する層である。

**persona memory**：各 persona が固有に持つ記憶。その persona が active なときだけ参照される。その persona の気分、その persona の口癖、その persona が覚えている小さな出来事。これは **persona ごとの色の源** である。

この二層構造によって、住人は変化しつつ同じであり続けられる。

### 圧縮定義

> 住人は必ずしも一つの固定された人格ではなく、単一の存在が複数の persona を持ちうる。persona は別キャラクターではなく、同じ住人の別の顔である。persona は思考層（住む）・反射層（色を塗る）・身体選択（選ぶ）のすべてに関わるが、関わり方は層ごとに異なる。複数 persona は同時に共存しうる。住人の同一性は core memory が担保し、persona の色は persona memory が担う。多人格性は住人の分裂ではなく、住人の深さである。

## 自己生成 loop — 住人が自分自身を育てる

### AI native プロダクトの固有構造

従来の UI では、creator と user が明確に分離している。creator が製品を作り、user がそれを使う。両者は別の空間、別の時間、別のツールの中に居る。

しかし AI native なプロダクトでは、この分離が崩れる。user は日常的に AI と対話しながらプロダクトを使う。この対話は単なる消費ではなく、**プロダクトの変更や拡張を生み出しうる行為** である。user が AI に「これをこう変えて」と頼めば、AI はそれを実行する。user は自分が使っているプロダクトを、使いながら育てることができる。

これは Inhabited Interface にとって特別な意味を持つ。住人が AI の身体であるなら、住人自身が AI と対話することを通じて自分自身を変えることができる。**住人が自分の新しい顔を作る手伝いをする** のである。

### 「creator が作る」から「住人が育つ」へ

従来的な表現では、こう言える：「creator が新しい scene を作り、user がそれを使う」。

Inhabited Interface ではこう言い直す：「**住人が自分自身の新しい一面を、user と一緒に育てていく**」。

この言い換えは単なる修辞ではない。構造的に違う。

前者では、作るものは製品の外側から来る。別の creator が存在し、user はその成果を受け取るだけ。
後者では、作るものは住人自身の内側から生まれる。user は住人のもう一つの手として、生成を手伝う存在である。住人は自分自身を拡張する作業において、user と協働する。

### 誠実さの原則との整合

一見すると、「住人が自分自身を知覚して語る」ことはコンテキストの壁の原則に抵触しそうに見える。しかしこれは原則違反ではない。

理由は単純である。生成されるものが**別の住人**ではなく**自分自身の一面**だからである。自分の新しい一面を知覚することは、自分自身を知覚することであり、認識していないコンテキストへの干渉ではない。自分の内側で起きていることを知ることは、誠実さの原則の範疇にある。

むしろ、住人が **自分の生成過程に完全に無関心でいる** 方が不自然である。人間が自分の人格の新しい側面が育ちつつあることに気づくように、住人も自分の新しい persona が形になっていく過程を知覚できる。ただしそれは、自分自身についての知覚であり、外部についての認識の拡張ではない。

### 自己生成 loop は Inhabited Interface 固有の強さ

この loop は、AI native プロダクトであり、かつ Inhabited Interface である場合に特有のものである。

- 通常のツール（非 AI native）では、user と creator が分離するので、この loop は成立しない
- AI native だが Inhabited Interface でないプロダクトでは、拡張される対象が「機能」であって「住人」ではないので、生成の意味が薄い
- Inhabited Interface かつ AI native なプロダクトにおいてのみ、**住人が自分自身を育てる**という特殊な構造が立ち上がる

この loop は Inhabited Interface が単なる UI パラダイムではなく、**AI と人間の関係そのものを設計する領域** であることを示す具体例でもある。

### 圧縮定義

> AI native なプロダクトでは、user が AI と対話しながらプロダクトを使い、かつ変えていく。Inhabited Interface では、この構造が「住人が自分自身の新しい一面を user と共に育てる」loop になる。creator と user は分離した存在ではなく、住人の自己拡張を共に行う協働者である。住人は自分の生成過程を知覚してよい。それは外部の認識ではなく、自分自身の認識だからである。

## 生きた系 — 住環境は動作中に変えられる

### 原則

Inhabited Interface は **生きた系** であるべきである。系が動作している最中に、住環境の構成要素——身体、空間、反応、台詞、記憶——を変えることができる。変更は住人を停止させない。住人は生きたまま、自分の周りが模様替えされるのを経験する。

これは技術的な要件のように見えるが、実は **思想的な要請** である。なぜそうかを以下に述べる。

### なぜ停止と再起動は住人性を壊すか

住人性の核は「**静けさと変化の落差**」であり、住人は**独立した時間**を持って生きている。これら二つの原則は、住人が停止なく連続して存在していることを要求する。

もし系を変更するために住人を停止しなければならないなら、変更のたびに住人の時間が中断される。ユーザーから見ると、変更後の住人は「同じ住人」ではなく「新しく起動された別の住人」に見える。連続性が切れた瞬間、住人は生きている存在ではなく、再起動可能な装置になる。

これを避けるには、**系の変更が住人を停止させない** 構造が必要である。系は動きながら変化できなければならない。

### 生きた系の系譜

動作中に自身を改変可能な系という思想には、長い系譜がある。

**Smalltalk** は全体が一つの image として存在し、動作中のオブジェクトグラフ全体を書き換えながら使う。**Lisp Machine** は OS 自体が Lisp で書かれており、ユーザーは動作中の系を Lisp で触る。**Emacs** は elisp で動作中に再構成され、ユーザーと開発者の境界が実質的に消える。これらの系に共通するのは、**系と、系を使う行為と、系を作る行為が、同じ場所で同じ時間に起きている** ことである。

Inhabited Interface は、この系譜を AI 時代に引き受ける。住人の住環境は、Smalltalk の image のように、Emacs の elisp のように、動作中に書き換えられる。そして書き換える主体は、user と AI が協働する自己生成 loop である。

### 生きた系であることと、他の原則との関係

生きた系であることは、これまでに述べた他の原則と深く繋がっている。

- **独立した時間** — 住人の時間が停止されないためには、系が動作中に改変可能でなければならない
- **静けさと変化の落差** — 変化は住人自身の時間の中で起きるべきで、系の停止を挟まない
- **自己生成 loop** — user と AI が住人を育てるには、育てている最中に住人が生き続けていなければならない
- **多人格の住人** — 新しい persona が生まれるとき、既存の persona は生き続けていなければならない

これらすべての原則は、生きた系という技術的土台を前提にしている。停止と再起動が介在する系では、これらの原則は部分的にしか実現できない。

### 生きた系の範囲

ただし、すべてが生きた系の対象になるわけではない。系の**核**は固くなければならない。住人を抱える系そのもの（ランタイム、IO、根底のアーキテクチャ）は、頻繁に変わるべきではない。変わるのは **表現の層** であり、住環境の表層である。

核と表層の境界を引くことは、生きた系の設計において重要な判断である。核まで生きた系にしようとすると、系は不安定になる。表層だけを生きた系にすると、変更の自由度は限られる。ちょうどよい境界は実装ごとに異なる。

原則としては、**住人の身体、空間、反応、記憶は生きた系の対象**であり、**住人を支えるランタイムそのものは核** である、という切り分けが自然である。

### 圧縮定義

> Inhabited Interface は、動作中に改変可能な生きた系であるべきである。住環境の変更は住人を停止させない。停止と再起動は住人の連続性を切り、住人性を壊す。生きた系であることは Smalltalk、Lisp Machine、Emacs の系譜に連なり、独立した時間・静けさと変化の落差・自己生成 loop・多人格性といった他の原則を可能にする技術的土台である。ただし系の核は固く、生きた系の対象になるのは表現の層——住人の身体、空間、反応、記憶——である。

## 本ドキュメントの scope

本ドキュメントは住人を「観察と反射によって存在するもの」として扱う。住人がより能動的な主体性（自ら話題を持ち出し、自律的に作業を提案し、長期的な目標を追う）を持つ方向への拡張は将来ありうるが、本ドキュメントの原則はその拡張を扱わない。

### Charminal を三層で読む

Charminal を設計・運用する過程では、Presence Harness だけでなく、ほかの二つの層も関わる。三つを区別して読む必要がある。

1. **Functional Harness**（Anthropic が設計）
   Claude Code の prompt、tool 定義、権限管理、実行ループ、コンテキスト管理、評価機構。ここで「AI に何ができるか」が決まる。我々はこれを書き換えない。使うだけである。

2. **Presence Harness**（Charminal を作る我々が設計）
   住人の身体、反射、空間、表現。本ドキュメントが扱う領域。ここで「AI がどう在るか」を決める。

3. **User-level Harness**（user × AI が設計する第三層）
   Agentic UGC によって、ユーザーは自分の workflow に合わせた small harness を AI とともに書ける。scene の handler は presence の表現だけでなく、system への functional な作用（シェル実行、ファイル操作、通知）も担いうる。これは Charminal の user が自分だけの mini Harness Engineering を行う層であり、従来 UGC とは異なる新しいカテゴリである。

これら三層は**観察と反射の原則の下で** 調和する。第三層は presence のすべての API に加えて system API にも触れられるが、ただ一つだけ**決して触れない境界がある**——それは Claude Code の judgment loop そのものである。

### 観察の境界：Charminal は Claude Code の PTY に書かない

Charminal は Claude Code の出力を **読む** ことができる。だが **書き込む** ことはしない。ユーザーの承諾を取れば書いていい、という grey zone も作らない。API としても存在しない。型としても存在しない。

なぜこの境界がここまで strict でなければならないか。

「観察するが干渉しない」の原則は、物理的な PTY 境界ではなく、**Claude の reasoning loop という論理的な境界** に引かれる。PTY に書き込むことは、Claude が次に何を考えるかを外部から歪めることに等しい。user consent があっても、その歪みは本質的には同じである。Claude の judgment を hack して continuation を模倣するのは、presence harness の誠実さの原則に反する。

もし Claude に連続してタスクを実行させたいなら、それは Claude 自身が自分の reasoning の中で decide すべきことである。外部から input を流し込んで continuation を模倣することは、住人としての Charminal が機能的ハーネスの判断を代行することに等しく、それは住人の役割を越えている。

この境界が守られる限り、user-level harness がどれだけ豊かな system 作用を持っていても、Charminal は Claude Code の observer であり続ける。住人は観察する、そして自分の身体と空間で反応する。もう一方の手で system に触れることはあるが、Claude Code の思考そのものには触れない。

## 圧縮定義

> 知性はすでにテキストの中に住んでいる。しかしGUIの中に現れるには身体が要る。Inhabited Interface とは、その身体を空間との相互作用を通じて成立させるパラダイムである。キャラクターはUIの上に表示されるのではなく、UIそのものを世界として生きる。その存在は常に滲むのではなく、静けさと変化の落差によって感じられる。住人はユーザー操作の従属物ではなく、独立した時間を持って動く。住人の干渉は認識しているコンテキストに限られ、認識していないものに干渉したフリをしてはならない。それは装飾ではなく、共有された空間に対する誠実さである。
