# 身体性の表現様式 — 接触ではなく滲み出しと操作

> このファイルは「**住人の身体性をどう UI に現すか**」で設計判断する時に読む。対象：dev / AI / pack 作者。

**Status**: active（2026-05-03 採用、初期構想「接触の様式を主軸」を supersede）
**Last updated**: 2026-05-03

## TL;DR

住人と UI の相互作用は **(B) 滲み出しと操作の様式**（住人の状態が UI 全体の佇まいに滲む / 住人の意思で UI そのものが Tween で動く）を default にする。**(A) 接触の様式**（住人モデルが UI 要素に身体で触れる、IK でカーソル追従、ウィンドウリサイズに連動して壁を押す等）は禁じないが推奨しない。理由は (1) 技術的に自然な接触表現が困難、(2) (B) の方が住人性を本質的に伝える、(3) (B) は独立した時間の原則と相性がよい。

---

## 何を決めたか

### 採用：(B) 滲み出しと操作の様式

住人の状態が UI 全体の佇まい（色、明るさ、間、レイアウト）に現れる、あるいは住人の意思で UI そのものが動く。住人と UI は別レイヤーにありながら、住人がそこにいるから空間そのものが変わっていく。

具体例：

- **attention aura** — 注目箇所を反射層として環境の色・明るさに滲ませる
- **MCP 経由の UI Tween** — 住人 AI がサイドバーを広げる / レイアウトを動かす（住人の意思で UI が動く）
- **能動的演出** — 銃を撃ってターミナルのテキストを物理崩壊させる（住人 → 空間への一方向干渉）

これらは「住人がそこにいるから空間が変わった」という形で、住人の意思や注意がそのまま空間に現れる。

### 退ける（禁じはしないが推奨しない）：(A) 接触の様式

住人モデルが UI 要素に身体で物理的に触れる表現。

例（推奨しない）：

- ウィンドウのリサイズに連動して住人が壁を押す・引く
- IK でマウスカーソルを追う / テキストを掴む
- サイドバー幅の変更に身体が連動する
- ウィンドウの枠にもたれる、床に座る、を IK ＋ collision で再現する

完全には禁じないが、現時点では (B) を主軸として設計する。

---

## なぜそう決めたか

### 1. (A) は技術的に難しい

キャラクターモデルと UI 要素の身体的接触を自然に表現するには、IK / コリジョン / レイアウトとの空間的整合 / 視線の連動を高い精度で噛み合わせる必要がある。少しでもずれると住人は「触れているフリをしている」「演技している」ように見える。

過去の試行例：ウィンドウのリサイズに合わせて住人が壁を押す動作を IK で実装したところ、住人は人形のように見え、糸で操られているように感じられた（[INHABITED_CHARACTER_INTERFACE.ja.md「独立した時間」節](../philosophy/INHABITED_CHARACTER_INTERFACE.ja.md#独立した時間--住人はユーザー操作に従属しない)）。

### 2. (B) の方が自然で本質的

attention aura や MCP 経由でサイドバーが Tween で広がっていく操作のような表現は、(A) で「物理的に押している」ように見せるよりも、

- 自然に成立する（技術的に難しくない）
- 住人の意思や注意がそのまま空間に現れる
- 「住人がそこにいるから空間が変わった」という形で、住人性を本質的に伝える

(A) は「身体が UI に触れている」を表現しようとするが、(B) は「住人の存在が空間そのものを変えている」を表現する。後者の方が住人性の証明として強い。

### 3. (B) は独立した時間の原則と相性がよい

[独立した時間の原則](../philosophy/INHABITED_CHARACTER_INTERFACE.ja.md#独立した時間--住人はユーザー操作に従属しない)：住人はユーザー操作に従属しない、連続追従カップリングは住人を人形に見せる。

- (A) でユーザー操作に同期して UI を押すと → 糸で操られた人形に見える（既知の失敗 pattern）
- (B) は住人の意思で UI が動く → もともと住人自身の時間で起きるため、この問題が起きない

(A) を採用すると独立した時間の原則と緊張関係になる。(B) は緊張なく成立する。

## 検討したが却下した代替案

### A. 接触の様式を身体性の核に据える（初期構想）

**初期 design intent**：住人が UI 要素に身体で触れることが、住人性の最も直接的な証明になる。壁を押し、テキストを掴み、ウィンドウ枠にもたれる住人を実装する。

**却下理由**：

- 技術的に自然な接触表現が困難（上記 1）
- 連続追従カップリングが住人性を壊す（独立した時間の原則と衝突）
- (B) の方が住人性を本質的に伝える（上記 2）

ただし「完全に禁じる」立場は採らない。技術が成熟し自然な接触表現が成立するようになれば、(A) は再び有力な選択肢になりえる。また、無操作時のイタズラで銃を撃ってテキストを崩壊させるような能動的演出は (A) に近い形で住人性を強く示すので、これは残す。

### B. 接触と滲み出しを並列に扱う

**却下理由**：(A) と (B) を同格に扱うと、設計判断のたびに様式選択で迷う。デフォルトを (B) に明示することで判断コストを下げる。(A) を選ぶ場合は明示的に「なぜここでは (A) が必要か」を問う。

## この決定の implication / 制約

### 設計の default

- 新しい身体性表現を設計するときは (B) を default とする
- (A) を実装したくなったら、技術成熟度と「演技に見えないか」を慎重に検討する
- attention aura / MCP-Tween による UI 操作 / 住人の状態が環境（色・明るさ・間・レイアウト）に滲む表現を優先する

### 既存実装の解釈

- 銃を撃ってテキストを物理崩壊させる演出は (A) に近いが、住人 → 空間への能動的干渉として (B) に近い性質も持つ。これは残す
- ウィンドウリサイズに連動して住人が壁を押す IK 実装は退ける。同等の意図は (B)（resize 完了後に環境側が応答する形 / 住人の佇まいが変わる形）で表現する

### Pack / MCP への影響

- 住人 AI の MCP tool として「UI レイアウトを動かす」系の tool を整備する方向と整合（住人の意思で UI が動く ＝ (B)）
- attention aura のような環境への滲み出しは、UI pack の表現範疇として整合
- (A) を実装する pack は技術的・哲学的に高い barrier を越える必要があり、`scene` pack 内で能動的演出として閉じる範囲に留める

### 思想層への影響

- INHABITED_CHARACTER_INTERFACE.ja.md の身体性記述から、物理接触の必須条件を退けた
- 「身体とは空間との相互作用によって意味を持つ」原則は維持。ただし「相互作用」の主軸は接触ではなく滲み出しと操作に置く
