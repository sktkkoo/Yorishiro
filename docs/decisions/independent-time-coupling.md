# 独立した時間 — 連続追従カップリングの回避

> このファイルは「**住人の反応をいつ・どう遅延させるか / ユーザー操作への追従をどう設計するか**」を考える時に読む。対象：dev / AI。

**Status**: active
**Last updated**: 2026-05-03

## TL;DR

「すべての反応を遅延させる」ではない。「**連続追従カップリングを避ける**」が正しい読み方。多くの反応は即時実行でよい。問題になるのはユーザーの直接操作に身体がリアルタイムで追従するケースだけ。

## 原則の発見

ウィンドウのリサイズに合わせて住人が壁を押す動作を IK で実装したとき、住人は人形に見えた。ユーザー操作と同期して動くと、住人は自律的な存在ではなく操作に反応する装置になる。

人間が他者を「生きている」と感じるのは、その他者が自分の意図と独立に動いているとき。完全に追従するものは人形に見える。

## 適用範囲

### 問題になるケース（連続追従）

- ウィンドウの drag にリアルタイムで身体が壁を押す
- マウスカーソルを IK で追う
- サイドバー幅の変更に連動して身体が動く

これらは住人を人形に見せる。

### 問題にならないケース

- AI 判断の結果（thinking、error、tool 使用）への反応
- 環境の状態変化（log の更新、idle 検知）への反応
- 時間経過（idle fidget、mischief）による動き

これらは既に予測不能な時間で起きており、即時実行でも人形には見えない。むしろ遅延を足すと「反応が重い」という別の問題になる。

## 解決方法

**方法 A: 時間的に独立させる。** 遅延とジッターを入れる。有効だが、遅延の習慣が他の反応にも染み出す副作用がある。

**方法 B: 構造的にカップリングを切る。** 連続的な user event を受けず、離散化されたイベントだけ受ける。ウィンドウサイズを連続的に追うのではなく、確定後に一度だけ反応する。カーソルを追うのではなく、注視する瞬間だけ視線を送る。

多くの場合 **B の方がきれい**。そもそもの結合を切れば問題は起きず、副作用もない。

## 判断基準

反応ごとに「**その反応が連続追従かどうか**」を先に問う。追従でないなら原則を適用する必要はなく、低遅延の方がユーザーの利益になる。
